# Supervisor Review Findings (2026-05-16)

Scope: Supervisor V2, orchestrator, state_machine.

## High
    1. [x] OrchestrationOrder and OrchestrationResponse consume methods clear only the pending flag under lock, but the fields are read afterward without a lock. A concurrent post can overwrite expectedBits/deadline/result while the consumer uses them, yielding inconsistent waits or responses. Refs: [src/supervisor/orchestrator.h](src/supervisor/orchestrator.h#L28-L82), [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L98-L167). Repro: trigger back-to-back state requests so the worker is waiting on an order while the state machine posts a new one; observe mismatched expectedBits/deadline or response data.
    2. [x] Design regression: consume() methods should copy all fields under spinlock (like ComponentMailbox::consumeNextState at [src/component_types.h](src/component_types.h#L64-L74)), but current code releases the lock then reads fields outside the critical section. Refs: [src/supervisor/orchestrator.h](src/supervisor/orchestrator.h#L44-L82), design spec [docs/superpowers/specs/2026-05-14-state-transition-impl-design.md](docs/superpowers/specs/2026-05-14-state-transition-impl-design.md#L94-L126). Repro: state machine calls consume() and reads orderMailbox_.expectedBits after lock is released; concurrent post() overwrites the value mid-read.
    3. [x] Mixed time units and underflow: startOrchestration stores xTaskGetTickCount() (ticks) plus getTransitionTimeout() (ms) into deadlineMs, while the worker calculates waitTicks using pdMS_TO_TICKS(deadlineMs - now). If deadlineMs < now, the subtraction underflows to a huge wait. Refs: [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L65-L146). Repro: set a very small timeout and start an orchestration after ticks advance; waitTicks wraps to a very large value and the worker blocks far longer than intended.
    4. [x] EventGroup bits are cleared after writing component mailboxes. A fast component can set its bit before the clear, causing a lost signal and a false timeout. Refs: [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L70-L85). Repro: have a component set its completion bit immediately when it sees the mailbox update; the bit is then cleared and the orchestration times out.
    5. [x] SupervisorV2 members are not initialized by the default constructor (observedState_, targetState_, nextState_, lastTargetBeforeError_, etc.). This can yield undefined behavior in run(), setTargetState(), and recovery logic. Refs: [src/supervisor/supervisor_v2.cpp](src/supervisor/supervisor_v2.cpp#L1-L4), [src/supervisor/supervisor_v2.h](src/supervisor/supervisor_v2.h#L329-L359). Repro: instantiate SupervisorV2 and call run() without explicitly setting observedState_/targetState_; behavior depends on indeterminate member values.
    6. [x] TaskHandle NULL dereference: postStateRequest() and postErrorEvent() call xTaskNotifyGive(supervisorTaskHandle_) without guarding against pre-setup() calls. supervisorTaskHandle_ is only set in setup() ([src/supervisor/supervisor_v2.cpp](src/supervisor/supervisor_v2.cpp#L9)), but components may register and post requests during their own setup before SupervisorV2::setup() runs. Refs: [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L39-L58). Repro: call postStateRequest() from a component init before supervisor setup(); FreeRTOS dereferences NULL or garbage handle.
    7. [x] stateRequestMailbox_ and errorEvent_ use spinlock-guarded post/consume but the consume methods do not copy fields under lock (unlike ComponentMailbox). Reader must lock again to safely read the fields after consume() clears pending. Refs: [src/supervisor/supervisor_v2.h](src/supervisor/supervisor_v2.h#L97-L120), compare to [src/component_types.h](src/component_types.h#L64-L74). Repro: consumeStateRequest() reads stateRequestMailbox_.requestedTarget outside any lock after consume() returns; concurrent postStateRequest() overwrites it.
     8. [x] setObservedState() and setTargetState() perform unguarded multi-field updates (observedState_, targetState_, lastTargetBeforeError_) without spinlock protection. A concurrent postStateRequest() or consumeStateRequest() can race and yield inconsistent state snapshots. Refs: [src/supervisor/state_machine.cpp](src/supervisor/state_machine.cpp#L183-L209). Repro: call postStateRequest() while run() is executing setTargetState(); lastTargetBeforeError_ is read and written in non-atomic fashion. Design decision: single-task state machine, no concurrent access.
    9. [x] Polling busy-loop instead of blocking: orchestrationWorker polls orderMailbox_ every 10ms via vTaskDelay instead of blocking on a semaphore signal from the state machine. This wastes CPU cycles and increases latency. Refs: [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L131-L139). Repro: monitor CPU usage; the worker task wakes 100 times per second even when idle.
    10. [x] EventGroup handle NULL check missing: setup() calls xEventGroupCreateStatic() without checking if eventGroup_ is successfully initialized. A hardware or memory error would go unnoticed. Refs: [src/supervisor/supervisor_v2.cpp](src/supervisor/supervisor_v2.cpp#L5-L6). Repro: mock xEventGroupCreateStatic to return NULL; eventGroup_ is never validated and subsequent xEventGroupSetBits crashes.
    11. [x] componentStatuses_ never reset to COMMITTED: once a component transitions to DEGRADED or FAILED, it remains stuck in that state for all future orchestrations. There is no reset mechanism or recovery path. Refs: [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L34), [L114-L117](src/supervisor/orchestrator.cpp#L114-L117), [src/supervisor/supervisor_v2.h](src/supervisor/supervisor_v2.h#L137-L142). Repro: trigger an error on an optional component; mark it DEGRADED; no future orchestration will include it even if the error is recovered.
    12. [x] TickType_t wrap-around on deadline calculations: fatalDeadlineMs = xTaskGetTickCount() + 60000 and orderMailbox_.post(... + getTransitionTimeout()) can wrap to small values if xTaskGetTickCount() is near UINT32_MAX. The deadline becomes invalid and timeouts fire immediately or never. Refs: [src/supervisor/state_machine.cpp](src/supervisor/state_machine.cpp#L231), [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L89). Repro: advance system ticks near UINT32_MAX (~49 days runtime) and trigger an orchestration; wait calculations overflow.
    17. [x] setup() NULL-check returns early without initializing supervisorTaskHandle_ and workerTaskHandle_. If eventGroup creation fails, these handles remain uninitialized and subsequent calls to orchestrationWorker() and state machine xTaskNotifyGive() dereference garbage. Refs: [src/supervisor/supervisor_v2.cpp](src/supervisor/supervisor_v2.cpp#L5-L20). Repro: mock xEventGroupCreateStatic to return NULL; eventGroup_ is nullptr, setup() returns early, later xEventGroupWaitBits(nullptr, ...) crashes in the worker. **Status:** Fixed — task handles are value-initialized to nullptr in header (L366-L367), setup() returns early before xTaskCreatePinnedToCore, xTaskNotifyGive calls are guarded, and xEventGroupClearBits in startOrchestration is now guarded (orchestrator.cpp:82-84), closing the native-test edge case.
    18. [x] nextState_ is unguarded: ActiveTransition struct (transitionTarget, subState) is read/written without spinlock across postNextComponentState ([L11](src/supervisor/orchestrator.cpp#L11)), startOrchestration ([L72](src/supervisor/orchestrator.cpp#L72)), and checkOrchestrationResponse ([L121](src/supervisor/orchestrator.cpp#L121)). On same-core preemption during mailbox write, the state machine can overwrite transitionTarget while the worker reads it. Refs: [src/supervisor/supervisor_v2.h](src/supervisor/supervisor_v2.h#L331), [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L4-L121). Repro: run at high priority and preempt postNextComponentState mid-function; nextState_.transitionTarget is overwritten while being read. Design decision: nextState_ accessed only by single state-machine task; workers read via spinlock-guarded mailboxes.
    19. [x] xEventGroupSetBits and xEventGroupClearBits calls lack NULL guard. If eventGroup_ is nullptr (post-setup failure), completeTransition and startOrchestration crash. Refs: [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L20), [L35](src/supervisor/orchestrator.cpp#L35), [L79](src/supervisor/orchestrator.cpp#L79). Repro: mock xEventGroupCreateStatic to fail; trigger an orchestration and a component tries to set/clear bits on nullptr. **Status:** Fixed — xEventGroupClearBits in startOrchestration is now guarded with `if (eventGroup_ != nullptr)` at orchestrator.cpp:82-84.

## Medium
    13. [x] Required component failure posts an error event but does not abort the active orchestration. The state machine stays in the orchestration path until the worker returns (likely timeout), delaying error handling. The comment claims it aborts. Refs: [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L24-L36), [src/supervisor/state_machine.cpp](src/supervisor/state_machine.cpp#L110-L121). Repro: during orchestration, have a required component report Failed; the supervisor keeps waiting for the worker response instead of aborting immediately.
    14. [x] postStateRequest() and postErrorEvent() call xTaskNotifyGive(supervisorTaskHandle_) even if setup() has not run and the handle is uninitialized. This is undefined on FreeRTOS. Refs: [src/supervisor/orchestrator.cpp](src/supervisor/orchestrator.cpp#L39-L58), [src/supervisor/supervisor_v2.cpp](src/supervisor/supervisor_v2.cpp#L5-L10). Repro: call postStateRequest() before setup(); FreeRTOS receives a notify on a garbage task handle.
    15. [x] getNextState() comments that ERROR/FATAL targets are immediate, but run() will still orchestrate toward ERROR/FATAL via stepTowardTarget() and startOrchestration(). That can delay or block error escalation. Refs: [src/supervisor/state_machine.cpp](src/supervisor/state_machine.cpp#L18-L76), [src/supervisor/state_machine.cpp](src/supervisor/state_machine.cpp#L110-L121). Repro: set targetState_ to ERROR while not already in ERROR; run() begins orchestration and waits for components instead of switching immediately.

## Low
    16. [x] checkComponentPresence() is implemented but never called, so missing required components are never flagged. Refs: [src/supervisor/state_machine.cpp](src/supervisor/state_machine.cpp#L54-L59), [src/supervisor/supervisor_v2.h](src/supervisor/supervisor_v2.h#L323-L327). Repro: leave a required component unregistered and start orchestration; no error event is posted.
