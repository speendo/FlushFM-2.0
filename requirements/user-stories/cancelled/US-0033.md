# US-0033: State-Management Pattern Compliance Audit

> **Status:** Cancelled — superseded by US-0044. V2 satisfies the spec's core requirements (RetryPolicy, DEGRADED, single-slot Mailbox, ErrorEvent flag+payload, Core 0 exclusivity, per-component required/optional). Remaining structural gaps captured in US-0044. | **Priority:** High | **Created:** 2026-05-09 | **Resolved:** 2026-05-17

## User Story
As a **maintainer**, I want **every requirement in the Supervisor Pattern to be verified against the codebase after US-0025 through US-0032 are finished**, so that **deviations from the approved state-management guideline are documented and every unfilled requirement has an explicit follow-up story**.

## Acceptance Criteria
- [ ] Each requirement in `state-management.md` §1 is checked against the codebase and marked satisfied, partially satisfied, or unsatisfied
- [ ] `TargetMode` and `ObservedState` exist as distinct variables; `ObservedState` advances only after all required components report `COMMITTED`
- [ ] Add `ComponentLifecycleStatus::Degraded` enum value for optional-component failures (see US-0025h)
- [ ] When optional component fails, lifecycle becomes `Degraded` (currently marked as `Failed` + `isDisabled`)
- [ ] Linear sequence `BOOTING → CONNECTING → READY → LIVE` is traversed strictly in order, including during recovery; `BOOTING` and `CONNECTING` are never a `TargetMode`
- [ ] `FATAL` state is implemented as a software-halted endpoint reachable from `ERROR` when `retryPolicy.isExhausted()`; no automatic recovery
- [ ] `RetryPolicy { maxRetries, recoveryCounter, isExhausted() }` struct exists scoped to the Supervisor
- [ ] `ERROR` recovery recalculates the highest justified `TargetMode` from input component states; does not restore the previous `TargetMode`
- [ ] Single-slot, last-write-wins Mailbox buffers incoming System Events; no deferred intent queue, no `isIntentEvent()`, no multi-slot FreeRTOS queue for events
- [ ] `SubState { PENDING, COMMITTED, FAILED }` enum is used per active transition; Supervisor tracks which individual components have reported per transition
- [ ] `ComponentStatus { COMMITTED, FAILED, DEGRADED }` map exists; optional component failures produce `DEGRADED` and are excluded from the quorum
- [ ] Absent optional components are treated as permanently `DEGRADED`; absent required components trigger immediate `ERROR` recovery
- [ ] Error Events use a single flag + payload `{ reason, source }`; first per transition only; cleared when Supervisor takes it up
- [ ] Component Commands route directly to the target component without passing through Supervisor
- [ ] Every component marked `required` or `optional` in configuration; component config struct includes `required: bool`
- [ ] Transition timeout configuration supports separate forward and backward timeout values per transition
- [ ] Components push state actively to Supervisor; Supervisor never polls component state
- [ ] Components implement at least `BOOTING`, `CONNECTING`, `READY`, `LIVE`, `SLEEP`, `ERROR`, `FATAL` states
- [ ] Each component defines a min/max state matrix indexed by `systemState` with a `SLEEP` row; backward transitions use `TARGET_MODE` dynamic reference in `maxState`
- [ ] SLEEP transition uses `PENDING → COMMITTED` sub-states; once in Deep Sleep no software runs
- [ ] Critical runtime data is persisted to NVS; NVS writes are limited to user-critical settings only
- [ ] Supervisor runs exclusively on Core 0; `TargetMode` and `ObservedState` are never written from Core 1
- [ ] `docs/` is updated to reflect any compliance gaps or implementation notes
- [ ] Debug and production builds succeed
- [ ] All native unit tests pass (`pio test -e native`)
- [ ] Any unsatisfied requirement is documented with a reference to either an existing open story or a recommended new follow-up story

## Notes
This is an audit-only story. No implementation is performed here. Each requirement from `state-management.md` §1 is checked against the codebase and marked satisfied, partially satisfied, or unsatisfied. Every gap is cross-referenced to an existing open story that already addresses it, or documented as a recommended new follow-up story.

### Expected satisfied by US-0025..US-0032
- US-0025d: Enum/string single source → `§Core Rules` enum definitions
- US-0025f/g/h: Event call-site migration, member cleanup, regression tests → `§Event System`
- US-0026c-f: Static ComponentId → `§Components` identity, `§Data Structures` component config
- US-0027a-c: Supervisor as module → `§Supervisor` Core 0, module boundary
- US-0028: targetState_ tracking → superseded by US-0031
- US-0029: Rename to Supervisor → naming consistency
- US-0030: EventPolicy rename → `§Event System`
- US-0031: TargetMode + ObservedState separation → `§Supervisor` two values
- US-0032: STATE_REQUESTED event → `§Event System`

### Likely gaps needing new follow-up stories
- FATAL state, RetryPolicy, and ERROR recovery recalculation (no existing story covers these)
- Single-slot Mailbox replacing FreeRTOS queue (US-0025e names it but doesn't implement the slot change)
- Formal SubState enum per transition
- Component DEGRADED status and absent-component handling
- Error Event flag + payload
- Component min/max state matrix with SLEEP row
- SLEEP sub-states during transition in
- Transition timeout config per direction
- ComponentBase interface with transitionTo/reportCommitted/reportFailed

### Known deviations from US-0026a spec
- `beginComponentTransition(ComponentID)` kept public (not removed as private) — `test_transition_completion_tracking` and `test_state_transition_flow` call it directly. Making it private requires test migration. No functional impact.
- Null-callback guard in `setComponentTransitionHooks(ComponentID)` still present (`if (!transitionInvoker \|\| !timeoutHook) return false`) — defensive check, not a runtime configuration path. Harmless.
- 4 test suites (`test_component_registry`, `test_transition_arbitration`, `test_orchestration_completion_policy`, `test_transition_timeout_watchdog`) still reference `SystemController` (old name) — pre-existing errors from US-0029, not introduced by this migration.
