# US-0032: Add STATE_REQUESTED Event Type to Minimal Event Contract

> **Status:** Cancelled — V2's `postStateRequest(target)` + `postErrorEvent(reason, source)` already provide a clean request/error API without a formal SystemEvent enum. A 2-event enum wrapper adds ceremony without changing behavior. | **Priority:** Medium | **Created:** 2026-05-08 | **Resolved:** 2026-05-17

## User Story
As a **maintainer**, I want **a minimal generic event contract with STATE_REQUESTED event**, so that **external state requests use a single canonical event type with target-state payload instead of specific event types per state**.

## Acceptance Criteria
- [ ] A new `SystemEvent::STATE_REQUESTED` event type is added to the SystemEvent enum
- [ ] `STATE_REQUESTED` carries a target `SystemState` (or `TargetMode`) as payload
- [ ] Event contract is documented: external requests use `STATE_REQUESTED(target)`, internal outcomes use `STATE_ENTERED` and `STATE_FAILED`
- [ ] Mailbox or event queue structure is updated to optionally carry state payload (or a separate payload struct)
- [ ] Existing state-request events (e.g., `PLAY_REQUESTED`, `STOP_REQUESTED`, `ENTER_SLEEP`) are reviewed for consolidation
- [ ] `handleEvent()` logic is updated to process `STATE_REQUESTED` with target-state matching
- [ ] Backward compatibility is maintained: old event types continue working or are deprecated gracefully
- [ ] Component event emissions remain unchanged (components still emit domain-specific signals, not `STATE_REQUESTED`)
- [ ] Tests validate that `STATE_REQUESTED` properly routes to target state transitions
- [ ] Debug and production builds succeed without errors
- [ ] All native unit tests pass (`pio test -e native`)

## Notes
- This completes the minimal event contract described in US-0025a
- Current event model uses specific events (PLAY_REQUESTED, STOP_REQUESTED); consolidation to `STATE_REQUESTED` is optional
- Payload handling requires careful consideration of event queue structure (current: single-slot Mailbox)
- Component-side signals remain domain-specific; this change is Supervisor-level only
- Consider whether `STATE_REQUESTED` should replace user-intent events entirely or coexist as an alternate interface
