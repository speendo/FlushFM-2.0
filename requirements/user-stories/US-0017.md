# US-0017: Component Architecture – Completion-Reporting Callback in Components

> **Status:** To Do  
> **Priority:** High  
> **Created:** 2026-03-29

---

## User Story

As a **component** (especially async ones like WiFi, Audio),  
I want **to report when my requested state change is completed asynchronously**,  
so that **the controller doesn't block and doesn't need to poll**.

---

## Acceptance Criteria

- [ ] SystemController provides a single completion callback interface: `reportCompletion(componentName, transitionId, status, debugReason)`
- [ ] `setXXX(transitionId)` returns timeout in milliseconds (`uint32_t`) and starts transition work
- [ ] Completion callback is required after every `setXXX()` call
- [ ] Callback signature: `void reportCompletion(const char* componentName, uint32_t transitionId, TransitionStatus status, DebugReason reason)`
- [ ] Controller matches callback reports to the pending transition by `transitionId`
- [ ] `transitionId` is passed to the component at the start of `setXXX()`
- [ ] Timeout hook contract: component implements `onTransitionTimeout(uint32_t transitionId)`
- [ ] Components guard against invalid `transitionId` and double-reporting
- [ ] WiFi component: returns timeoutMs, then reports `Completed` or `Failed` asynchronously
- [ ] Audio component: returns timeoutMs, then reports `Completed` or `Failed` asynchronously
- [ ] Synchronous components (e.g., BoardInfo): return timeoutMs and report `Completed` immediately in the same tick
- [ ] All components have at least stub implementations
- [ ] No compilation errors

---

## Test Strategy

**Write tests first (TDD per `testing.md`):**
- [ ] Unit test: Component calls `reportCompletion(name, transitionId, Completed, nullptr)` → controller marks transition done
- [ ] Unit test: Component calls `reportCompletion(name, transitionId, Failed, reason)` → controller applies required/optional policy
- [ ] Unit test: Component receives `onTransitionTimeout(transitionId)` → moves to safe state → emits `reportCompletion(..., Failed, "timeout")`
- [ ] Unit test: Late callback with stale `transitionId` is silently ignored
- [ ] Unit test: Double-reporting same transition (two callbacks for same `transitionId`) → second ignored
- [ ] Unit test: Synchronous component (BoardInfo) returns timeout and emits `Completed` in same tick
- [ ] Unit test: Asynchronous component (WiFi) returns timeout and emits callback later via task/deferral

**Corner Cases & Runtime Safety:**
- [ ] Unit test: `reportCompletion(nullptr, transitionId, ...)` rejected or handled safely (no null dereference)
- [ ] Unit test: `reportCompletion("", transitionId, ...)` (empty name) handled consistently
- [ ] Unit test: Component name > 256 chars does not overflow internal buffers
- [ ] Unit test: `reportCompletion(name, transitionId=0, ...)` accepted (valid edge case)
- [ ] Unit test: `reportCompletion(name, transitionId=UINT32_MAX, ...)` accepted without overflow
- [ ] Unit test: `reason` string > 512 chars does not overflow or truncate dangerously
- [ ] Unit test: `reason = nullptr` accepted and handled (reason is optional)
- [ ] Unit test: Multiple components reporting callbacks simultaneously → no race condition in controller queue
- [ ] Unit test: Component calls `reportCompletion()` from interrupt context → controller handles safely (no deadlock)
- [ ] Unit test: Component calls `reportCompletion()` synchronously during `setXXX()` → no re-entrancy crash
- [ ] No memory leak if callback contains large temporary strings/buffers
- [ ] No stack overflow from deeply nested callback chains

Tests use mock components and controller; run on host via `[env:native]`.

---

## Notes

Single contract:
- `setXXX(transitionId)` starts the transition and returns timeoutMs.
- Exactly one later `reportCompletion(...)` is expected for every call.
- On timeout, SystemController calls `onTransitionTimeout(transitionId)`.
- Timeout handler must move component to safe state and then emit `reportCompletion(..., Failed, reason)`.
- Late callbacks for stale transition IDs are ignored.

This keeps synchronous and asynchronous components on one shared protocol while avoiding split code paths.
