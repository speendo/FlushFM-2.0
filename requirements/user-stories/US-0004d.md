# US-0004d: Runtime Status APIs for WiFi and Audio

> **Status:** To Do  
> **Priority:** Medium  
> **Created:** 2026-03-28

---

## User Story

As a **developer**,  
I want **WiFi and Audio components to expose read-only runtime status**,  
so that **orchestration and transitions do not depend on implicit flags or log side effects**.

---

## Acceptance Criteria

- [ ] WiFi component exposes an explicit WiFi state model (`DISCONNECTED`, `CONNECTING`, `CONNECTED`, `ERROR`)
- [ ] Audio component exposes a read-only runtime state suitable for orchestration decisions
- [ ] Existing boolean-only connection checks are replaced or backed by explicit state access
- [ ] Orchestration logic uses component status APIs for transition decisions
- [ ] No component exposes mutable internal state directly
- [ ] Build and current manual CLI scenarios continue to work

---

## Notes

- Keep API surface minimal and focused on orchestration needs.
- This story enables deterministic transition guards in the system controller.
- Suggested implementation sequence:
	1. Define `WiFiState` and audio runtime-state enums with read-only getters.
	2. Back existing boolean checks with explicit state mapping.
	3. Switch orchestration transition decisions to status APIs and validate CLI flows.
