# US-0017: Component Architecture – Completion-Reporting Callback in Components

> **Status:** Done  
> **Priority:** High  
> **Created:** 2026-03-29

---

## User Story

As a **component** (especially async ones like WiFi, Audio),  
I want **to report when my requested state change is completed asynchronously**,  
so that **the controller doesn't block and doesn't need to poll**.

---

## Acceptance Criteria

- [x] SystemController provides a single completion callback interface: `reportCompletion(componentName, transitionId, status, debugReason)`
- [x] `setXXX(transitionId)` returns timeout in milliseconds (`uint32_t`) and starts transition work
- [x] Completion callback is required after every `setXXX()` call
- [x] Callback signature: `void reportCompletion(const char* componentName, uint32_t transitionId, TransitionStatus status, DebugReason reason)`
- [x] Controller matches callback reports to the pending transition by `transitionId`
- [x] `transitionId` is passed to the component at the start of `setXXX()`
- [x] Timeout hook contract: component implements `onTransitionTimeout(uint32_t transitionId)`
- [x] Components guard against invalid `transitionId` and double-reporting
- [x] WiFi component: returns timeoutMs, then reports `Completed` or `Failed` asynchronously
- [x] Audio component: returns timeoutMs, then reports `Completed` or `Failed` asynchronously
- [x] Synchronous components (e.g., BoardInfo): return timeoutMs and report `Completed` immediately in the same tick
- [x] All components have at least stub implementations
- [x] No compilation errors
- [x] For audio stop transitions, `reportCompletion(..., Completed, ...)` means streaming has actually stopped, not just that stop was requested

## Test Strategy

**Write tests first (TDD per `testing.md`):**
- [x] Unit test: Component calls `reportCompletion(name, transitionId, Completed, nullptr)` → controller marks transition done
- [x] Unit test: Component calls `reportCompletion(name, transitionId, Failed, reason)` → controller applies required/optional policy
- [x] Unit test: Component receives `onTransitionTimeout(transitionId)` → moves to safe state → emits `reportCompletion(..., Failed, "timeout")`
- [x] Unit test: Late callback with stale `transitionId` is silently ignored
- [x] Unit test: Double-reporting same transition (two callbacks for same `transitionId`) → second ignored
- [x] Unit test: Synchronous component (BoardInfo) returns timeout and emits `Completed` in same tick
- [x] Unit test: Asynchronous component (WiFi) returns timeout and emits callback later via task/deferral

**Corner Cases & Runtime Safety:**
- [x] Unit test: `reportCompletion(nullptr, transitionId, ...)` rejected or handled safely (no null dereference)
- [x] Unit test: `reportCompletion("", transitionId, ...)` (empty name) handled consistently
- [x] Unit test: Component name > 256 chars does not overflow internal buffers
- [x] Unit test: `reportCompletion(name, transitionId=0, ...)` accepted (valid edge case)
- [x] Unit test: `reportCompletion(name, transitionId=UINT32_MAX, ...)` accepted without overflow
- [x] Unit test: `reason` string > 512 chars does not overflow or truncate dangerously
- [x] Unit test: `reason = nullptr` accepted and handled (reason is optional)
- [x] Unit test: Multiple components reporting callbacks simultaneously → no race condition in controller queue
- [x] Unit test: Component calls `reportCompletion()` from interrupt context → controller handles safely (no deadlock)
- [x] Unit test: Component calls `reportCompletion()` synchronously during `setXXX()` → no re-entrancy crash
- [x] No memory leak if callback contains large temporary strings/buffers
- [x] No stack overflow from deeply nested callback chains

Tests use mock components and controller; run on host via `[env:native]`.

## Notes

Single contract:
- `setXXX(transitionId)` starts the transition and returns timeoutMs.
- Exactly one later `reportCompletion(...)` is expected for every call.
- On timeout, SystemController calls `onTransitionTimeout(transitionId)`.
- Timeout handler must move component to safe state and then emit `reportCompletion(..., Failed, reason)`.
- Late callbacks for stale transition IDs are ignored.
- For audio stop transitions, the completion callback is the proof that playback is really stopped and the controller may continue with later shutdown steps.

This timeout/callback contract is part of the later orchestration and shutdown flow; it is intentionally not part of the audio-only stop story in US-0007.

This keeps synchronous and asynchronous components on one shared protocol while avoiding split code paths.
