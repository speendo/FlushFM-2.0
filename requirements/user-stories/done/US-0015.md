# US-0015: Component Architecture – Component Lifecycle Registry in SystemController

> **Status:** Done  
> **Priority:** High  
> **Created:** 2026-03-29

---

## User Story

As a **system controller**,  
I want **to track which components are present, required, and healthy**,  
so that **I can make correct decisions about required vs optional failures during state transitions**.

---

## Acceptance Criteria

- [x] SystemController stores per-component: {name, lifeCycleStatus, isRequired, isDisabled, lastFailureReason}
- [x] Add method: `registerComponent(const char* name, bool isRequired)` with logging
- [x] Add method: `getComponentStatus(const char* name) → ComponentLifecycleStatus`
- [x] Add method: `markComponentFailed(const char* name, const char* reason)` with logging
- [x] Add method: `isComponentRequired(const char* name) → bool`
- [x] Required/optional is defined once per component in controller-owned configuration (not per target state)
- [x] Registry persists only for one boot (resets to Unknown on startup)
- [x] No hardcoding of component lists; registry is data-driven
- [x] Components can opt-in to register themselves
- [x] All existing components register themselves on startup
- [x] No compilation errors

## Test Strategy

**Write tests first (TDD per `testing.md`):**
- [x] Unit test: `registerComponent(name, isRequired)` stores registry entry and returns success
- [x] Unit test: `getComponentStatus(name)` returns correct status after registration
- [x] Unit test: `markComponentFailed(name, reason)` updates component state and logs failure
- [x] Unit test: `isComponentRequired(name)` returns correct required/optional classification
- [x] Unit test: Registry handles duplicate registration gracefully (skip or update)
- [x] Unit test: Unknown component queries return sensible default (Unknown status, required=false)
- [x] Unit test: Registry resets on boot (new SystemController instance has empty registry)

**Corner Cases & Runtime Safety:**
- [x] Unit test: `registerComponent(nullptr, true)` rejected or handled safely (no crash)
- [x] Unit test: `registerComponent("", true)` (empty string) rejected or handled safely
- [x] Unit test: `registerComponent()` with component name > 256 chars does not overflow buffer
- [x] Unit test: `markComponentFailed(valid_name, nullptr)` accepted (reason is optional)
- [x] Unit test: `markComponentFailed(valid_name, long_reason)` with reason > 512 chars does not overflow
- [x] Unit test: Registering 20 components does not crash map or cause memory issues
- [x] Unit test: Registry iteration is safe during concurrent modifications (if multi-threaded) - not applicable because registry is Core 0 exclusive
- [x] No use-after-free: component names stored as copies (not pointers to caller's stack)
- [x] No memory leaks when clearing/destroying registry
- [x] No null pointer dereference in `getComponentStatus()`, `isComponentRequired()` edge cases

Tests use mock components; no real hardware needed. Run on host via `[env:native]`.

---

## Notes

Registry uses `std::map<std::string, ComponentRegistryEntry>` internally.
Components register during startup via their own `registerWithController()` hook.
This is the foundation for required/optional policy in transitions.
