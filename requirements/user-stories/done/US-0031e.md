# US-0031e: Component min/max State Matrices

> **Status:** Done | **Priority:** High | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **components to define a min/max state matrix indexed by TargetMode**, so that **forward and backward transition timeouts are enforced per-component and the contract is explicit**.

## Acceptance Criteria
- [x] Each component defines a `minState`/`maxState` matrix indexed by `TargetMode` (or `SystemState`)
- [x] On forward transitions (moving to higher-ranked state), the `minState` is enforced with a forward timeout
- [x] On backward transitions (moving to lower-ranked state), the `maxState` is enforced with a backward timeout
- [x] `maxState` supports a dynamic `TARGET_MODE` reference that resolves to the current `targetMode_` at transition time
- [x] Forward and backward timeouts are configured separately per component
- [x] The matrix includes a SLEEP row (defines component behavior during transition into SLEEP, not during Deep Sleep)
- [x] Reference implementation exists for at least one component (e.g., WiFi) to demonstrate the contract
- [x] All existing tests pass; transition timeout tests continue to work
- [x] Debug and production builds succeed

## Notes

### Design — per-component structure

Each component exposes a static matrix:

```cpp
struct ComponentStateMatrix {
    TargetMode minState;  // enforced on forward transitions (timeout)
    TargetMode maxState;  // enforced on backward transitions (timeout)
};

// Special value: TARGET_MODE — resolves to current Supervisor targetMode_
constexpr TargetMode TARGET_MODE = static_cast<TargetMode>(0xFF);
```

Example for WiFi component:

```cpp
// WiFi state matrix:
//   In SLEEP: WiFi must be initialized (min=SLEEP), can't go above READY (max=READY)
//   In CONNECTING: WiFi must attempt connection (min=CONNECTING), max=READY
//   In READY: WiFi must be connected (min=READY), max=LIVE
//   In LIVE: WiFi stays connected (min=READY, actually LIVE in practice)
//   In ERROR: WiFi resets to BOOTING (min=BOOTING), max=BOOTING
//   In FATAL: WiFi halts (min=FATAL), max=FATAL
//   In BOOTING: WiFi initializes (min=BOOTING), max=CONNECTING

constexpr ComponentStateMatrix kWiFiStateMatrix[] = {
    /* FATAL      */ {TargetMode::FATAL, TargetMode::FATAL},
    /* ERROR      */ {TargetMode::BOOTING, TargetMode::BOOTING},
    /* BOOTING    */ {TargetMode::BOOTING, TargetMode::CONNECTING},
    /* SLEEP      */ {TargetMode::SLEEP, TargetMode::READY},
    /* CONNECTING */ {TargetMode::CONNECTING, TargetMode::READY},
    /* READY      */ {TargetMode::READY, TargetMode::LIVE},
    /* LIVE       */ {TargetMode::READY, TargetMode::LIVE},
};
```

### Integration with orchestration

In `beginOrchestration()`, when arming a component for transition:
1. Look up the component's state matrix entry for the `targetMode_` (target state)
2. If forward transition (`targetMode_ > observedState_`): enforce `minState` with forward timeout
3. If backward transition (`targetMode_ < observedState_`): enforce `maxState` with backward timeout
4. The `transitionInvoker` callback already returns a `uint32_t timeoutMs` — this can use the matrix values

### How current transition hooks work

Current signature:
```cpp
using TransitionInvoker = std::function<uint32_t(SystemState target, uint32_t transitionId)>;
// Returns timeoutMs
```

The invoker currently returns a hardcoded timeout (1000ms in tests). With the matrix:
1. On transition to `target`, look up `matrix[target].minState` (if forward) or `matrix[target].maxState` (if backward)
2. Component's invoker reads the required state and returns an appropriate timeout
3. If `TARGET_MODE` is the value, resolve it to the actual current `targetMode_`

### Forward vs backward detection

```cpp
bool isForward = targetRank > stateRank;
TargetMode requiredState = isForward 
    ? matrix[targetIdx].minState 
    : matrix[targetIdx].maxState;
```

### Timeout configuration

Forward timeout: how long the component has to reach its minState when advancing
Backward timeout: how long the component has when stepping down

These can be different — backward transitions (shutdown, stop) may need different timing than forward (startup, connect).

### Minimal first implementation

For US-0031e, implement the structure and reference implementation for WiFi. Audio follows the same pattern. CLI and BoardInfo can use trivial matrices (always satisfied). The enforcement mechanism can be a helper that `beginOrchestration()` calls.
