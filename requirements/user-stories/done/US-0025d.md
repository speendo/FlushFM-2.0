# US-0025d: Enum/String Single Source Mapping

> **Status:** Done | **Priority:** High | **Created:** 2026-04-25

## User Story
As a **maintainer**, I want **to remove duplicated enum-label maintenance**, so that **enum names and string labels are derived from one central source**.

## Acceptance Criteria
- [x] A single shared enum-to-string mapping mechanism is defined and approved (e.g., X-macro list, or central mapping table in a dedicated header)
- [x] The following enums are migrated to use the shared mechanism: (1) `SystemEvent`, (2) `SystemState`, (3) `EventPolicy`, (4) any other enums used in state/event handling
- [x] For each enum, a corresponding string-label function is generated/derived from the shared definition (e.g., `toString(SystemEvent)` derives from the same source as the enum itself)
- [x] No duplicate switch-based label tables remain in the codebase; all previous manual `toString()` implementations are removed and replaced by the shared mechanism
- [x] The shared mechanism is easy to extend: adding a new enum value requires editing only the shared definition, not multiple switch blocks
- [x] Debug and CLI output uses the shared mechanism for all enum labels (consistency across logging, CLI commands, diagnostic output)
- [x] All references to enum labels in comments, CLI help text, and documentation reference the shared mechanism (no hardcoded strings)
- [x] Unit tests verify enum-to-string mapping integrity: (1) round-trip conversion (enum → string → enum) for all values, (2) no gaps or duplicates, (3) label consistency across all output pathways
- [x] Debug build succeeds
- [x] Production build succeeds
- [x] No code-behavior changes; only refactoring of maintenance mechanism

## Notes
This story depends on contract decisions from US-0025c (event types, naming standards). It applies those decisions to the shared enum-to-string mapping.

Scope is maintenance reduction only; no new enum values introduced, no new control-flow paths added. The behavior of the system remains unchanged; only how enum names are maintained is simplified.

Verification:
- Search codebase for `toString(...)` implementations and switch-based label tables; verify all are removed after migration
- Build and run native unit tests to verify mapping integrity
- Spot-check debug output (Serial CLI, logs) for label consistency across different output points

### Macro design

Two patterns — one for enums with explicit values (`SystemState`), one for auto-increment enums.

**Enums with explicit values** — `SystemState` (`uint8_t`, non-contiguous):

```cpp
#define SYSTEM_STATE_X(V) \
    V(ERROR,      0) \
    V(BOOTING,   10) \
    V(SLEEP,     20) \
    V(CONNECTING, 30) \
    V(READY,     40) \
    V(LIVE,      50)

#define SYSTEM_STATE_DEF(n, v) n = v,
enum class SystemState : uint8_t { SYSTEM_STATE_X(SYSTEM_STATE_DEF) };
#undef SYSTEM_STATE_DEF

#define SYSTEM_STATE_STR(n, v) case SystemState::n: return #n;
constexpr const char* toString(SystemState s) {
    switch (s) { SYSTEM_STATE_X(SYSTEM_STATE_STR) }
    return "UNKNOWN";
}
#undef SYSTEM_STATE_STR
#undef SYSTEM_STATE_X
```

**Enums without explicit values** — `SystemEvent`, `EventPolicy`, `SystemReason`, `TransitionRequestDecision`, `TransitionStatus`, `ComponentLifecycleStatus`:

```cpp
#define SYSTEM_EVENT_X(V) \
    V(BOOT) V(COMPONENT_SETUP_FAILED) V(WIFI_READY) V(AUDIO_INIT_OK) \
    V(AUDIO_INIT_FAILED) V(PLAY_REQUESTED) V(STOP_REQUESTED) \
    V(WIFI_DISCONNECTED) V(STREAM_LOST) V(RECOVER) V(ENTER_SLEEP)

#define SYSTEM_EVENT_DEF(n) n,
enum class SystemEvent { SYSTEM_EVENT_X(SYSTEM_EVENT_DEF) };
#undef SYSTEM_EVENT_DEF

#define SYSTEM_EVENT_STR(n) case SystemEvent::n: return #n;
constexpr const char* toString(SystemEvent e) {
    switch (e) { SYSTEM_EVENT_X(SYSTEM_EVENT_STR) }
    return "UNKNOWN";
}
#undef SYSTEM_EVENT_STR
#undef SYSTEM_EVENT_X
```

Apply the same no-value pattern to:
- `SystemReason` (9 values: NONE, BOOT_SEQUENCE, COMPONENT_SETUP, WIFI_INITIALIZED, AUDIO_TASK_STARTED, AUDIO_TASK_INIT_FAILED, USER_REQUEST, RECOVERY, POWER_POLICY)
- `EventPolicy` (2 values: FIRE_AND_FORGET, BOUNDED_BLOCKING) — **new toString**, none exists today
- `TransitionRequestDecision` (4 values: Ignored, Started, Superseded, Queued)
- `TransitionStatus` (2 values: Completed, Failed)
- `ComponentLifecycleStatus` (4 values: Unknown, Ready, Failed, Disabled)
