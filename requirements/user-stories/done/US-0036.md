# US-0036: Event System Consolidation — Remove Legacy Events and Transition Queue

> **Status:** To Do | **Priority:** Medium | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **all state requests to use a single generic `STATE_REQUESTED` event type with no legacy-specific events and no transition queuing**, so that **the event system aligns with `state-management.md`'s single-slot Mailbox model and the no-queue policy**.

## Acceptance Criteria
- [ ] `PLAY_REQUESTED`, `STOP_REQUESTED`, `ENTER_SLEEP`, `RECOVER` removed from `SystemEvent` enum
- [ ] Only `COMPONENT_SETUP_FAILED` and `STATE_REQUESTED` remain in the enum
- [ ] All callers (CLI, debug CLI, main.cpp, tests) migrated to `STATE_REQUESTED(targetState)` via 3-param `postEvent`
- [ ] Internal `handleEvent()` auto-play calls use `STATE_REQUESTED` with `mailbox_.targetState` set first
- [ ] `handleEvent()` has no duplicated legacy if-chain and no empty `switch (observedState_)`
- [ ] `TransitionRequestDecision` enum removed entirely; `requestTransition()` returns `bool`
- [ ] `hasQueuedStateTransition_` / `queuedStateTransition_` removed from Supervisor
- [ ] All public queued-transition accessors (`hasQueuedTransition()`, `queuedTransitionId()`, etc.) removed
- [ ] `docs/event-contract-model.md` updated to reflect the 2-event enum
- [ ] Debug and production builds succeed without errors
- [ ] All native unit tests pass

## Notes
- This story completes the event consolidation started in US-0032 (STATE_REQUESTED added) and US-0035 (component commands removed)
- After this story, the full event contract is: `COMPONENT_SETUP_FAILED` (failure signals) and `STATE_REQUESTED` (all state intents via target payload)
- The `TransitionRequestDecision` enum becomes dead code after removing Queued — `Started` and `Superseded` both just mean "true" to the caller, so a `bool` return suffices
- The queued transition slot violated the no-queue policy from `state-management.md`
- Data structure alignment with `state-management.md` (§3 Reference Pattern) is out of scope — handled in a follow-up
