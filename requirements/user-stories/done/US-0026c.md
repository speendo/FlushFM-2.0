# US-0026c: Static ComponentId Introduction and Shim

> **Status:** Done | **Priority:** Medium | **Created:** 2026-04-25

## User Story
As a **maintainer**, I want **to establish a static component identity model with a controlled transition path**, so that **downstream migration can be deterministic and bounded**.

## Acceptance Criteria
- [x] Static identifier set is defined for runtime components
- [x] Registration supports identifier-based path
- [x] Transitional compatibility path is explicitly scoped and time-boxed

## Notes
This story introduces the static ComponentId model. Full API migration happens in US-0026d.

### Implementation
- `ComponentID` enum with `BoardInfo`, `WiFi`, `AudioRuntime`, `CLI`, `Count` in `component_types.h`
- `componentName(ComponentID)` returns human-readable diagnostic labels
- `ISystemComponent` constructor takes `(ComponentID id, const char* name)`; all 4 component constructors updated
- `Supervisor` gained `ComponentID` overloads for `registerComponent`, `getComponentStatus`, `isComponentRequired` — all delegate to existing string-based APIs
- Round-trip tests added in `test_component_types`
- Test mock in `test_system_components_contract` updated for new constructor signature
- String-based APIs remain untouched (removed in US-0026f)
- Internal maps remain `std::map<std::string, ...>` (migrated to `std::array` in US-0026d)
