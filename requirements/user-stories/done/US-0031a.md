### Why FATAL at value 0?

FATAL is the Level 0 inert state — software halted, below everything. Shifting ERROR to 5 places it just above FATAL but below BOOTING(10). The rank ordering becomes:

| Value | State |
|-------|-------|
| 0 | FATAL |
| 5 | ERROR |
| 10 | BOOTING |
| 20 | SLEEP |
| 30 | CONNECTING |
| 40 | READY |
| 50 | LIVE |

### FATAL behavior

No transition logic for FATAL is added in this story. Events arriving in FATAL state fall through the `switch (observedState_)` to the default case (no-op). This is correct: a halted system should process no events. Full FATAL wiring is in US-0031d.

### No test changes needed

Test files include `supervisor.cpp` directly via `#include "../../src/state_machine/supervisor.cpp"` — the rename propagates automatically. `state()` return value is unchanged. No test inspects private `state_` or `targetState_` by name.
