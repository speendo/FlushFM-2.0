## User Story

As a **system orchestrator**,  
I want **to call component state methods and wait for completions, respecting required/optional policies**,  
so that **the system handles partial failures gracefully without blocking or crashing**.

---

## Acceptance Criteria

- [x] SystemController internal state tracks: {currentTransition, targetState, componentsWaitingForCompletion}
- [x] Define a static required/optional classification per component (component-level, not per target state)
- [x] Transition flow implemented: call `component.setXXX(transitionId)` â†’ store returned timeoutMs â†’ wait for completion callbacks â†’ commit or revert
- [x] Completion callbacks are mandatory for all components after `setXXX()`
- [x] Handle callback `Completed`: mark component done
- [x] Handle callback `Failed`: apply required/optional policy
- [x] Policy for Required+Failed: do NOT commit target state; revert or enter ERROR state with logging
- [x] Policy for Optional+Failed: mark as degraded but continue; commit target state anyway
- [x] Use timeout returned by each `setXXX()` call as the watchdog value
- [x] Enforce timeout: if a component does not report completion before timeout, SystemController calls `component.onTransitionTimeout(transitionId)`
- [x] Timeout hook behavior: component enters a safe internal state and then reports `Failed` via callback (optional reason)
- [x] Each transition gets a unique transitionId for logging and matching completions
- [x] SystemController supports one in-flight transition per component; stale callbacks (old transitionId) are ignored
- [x] On timeout, SystemController does not start a new transition for that component until timeout recovery has completed
- [x] Log each state transition and result with transitionId for tracing in debug mode
- [x] Log failed and timeouts in debug mode and production mode
- [x] Test: manual transition test via serial CLI (e.g., send "transition idle")
- [x] All existing state transitions (OFFâ†’STARTINGâ†’IDLEâ†’STREAMINGâ†’ERROR) work with new orchestration
- [x] For stop/shutdown transitions, the controller advances only after the completion callback confirms the component has actually stopped streaming

## Notes

Required/Optional classification can be hardcoded for now in a static controller-owned config.
Example: REQUIRED = {WiFi, AudioRuntime}; OPTIONAL = {CLI, BoardInfo}
Timeout per component is returned by `setXXX(transitionId)` and treated as watchdog value.
On timeout, recovery is component-owned via `onTransitionTimeout(transitionId)` and completion callback.
No global timeout fallback; components MUST return valid timeout values for all target states.
This story owns the later timeout-driven orchestration for light-sensor/power-off sequencing; US-0007 stays limited to audio mute-first stop behavior.
