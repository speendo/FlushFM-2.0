# US-0007: Clean Audio Stop via Mute-First Shutdown

> **Status:** Done  
> **Priority:** High  
> **Created:** 2026-03-11

---

## User Story

As a **user**,  
I want **audio to become silent immediately when a stop is requested**,  
so that **music does not continue playing while the stream is being stopped**.

---

## Acceptance Criteria

- [x] `IAudioPlayer` exposes `setMute(bool)` and `getMute()`
- [x] `AudioPlayerESP32::stop()` mutes first and then calls `stopSong()`
- [x] `AudioPlayerESP32::connectToHost()` clears mute before starting playback
- [x] Repeated `setMute()` calls are safe and idempotent
- [x] No blocking wait or polling is added to the audio wrapper for stop handling

---

## Notes

**Root cause:** `audio.stopSong()` (ESP32-audioI2S) stops the decoder and HTTP client immediately, but the I2S DMA peripheral can continue draining already-filled hardware buffers independently of the CPU.

**Approach options:**
1. Mute first via `setMute(true)` and then call `audio.stopSong()` to make the output silent immediately
2. ~~Reduce DMA buffer size to minimise drain time~~ – **rejected**: unacceptable impact on dropout resilience at high bitrates
3. Use the I2S peripheral's `i2s_zero_dma_buffer()` IDF call to flush immediately
4. Set volume to 0 via `setVolume(0)` before stopping – takes effect immediately in the I2S output stage, but couples stop behavior to volume state and was therefore not chosen

Option 1 is the chosen implementation: instant silence, no IDF dependency, no polling in the audio wrapper. Option 3 remains as fallback if a hard DMA flush is ever needed.

**Observed behavior (US-0003):** `vTaskSuspend(audioTask)` causes audio to keep playing for several seconds – do not use suspend as a stop mechanism in production.

**Out of scope for this story:** relay control, sleep entry, and any timeout-based power-off sequencing belong to future user stories in the system-state / light-sensor flow (for example US-0015 to US-0020), not to the audio wrapper.
