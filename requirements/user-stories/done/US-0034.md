# US-0034: Replace FreeRTOS Event Queue with Single-Slot Mailbox

> **Status:** Done | **Priority:** High | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **the FreeRTOS event queue replaced with a single-slot last-write-wins Mailbox**, so that **the event flow matches the state-management.md pattern and the deferred-intent infrastructure is removed**.

## Acceptance Criteria
- [x] `Mailbox { SystemEvent event, SystemReason reason, bool pending }` struct replaces `QueueHandle_t queue_`, `deferredIntentEvents_`, `pendingCriticalEvent_`, `pendingEvent_`, and `pendingReason_`
- [x] `postEvent()` writes directly to the Mailbox slot (ARDUINO) or dispatches directly (native); no `xQueueSend`, no sticky fallback logic
- [x] `processEventQueue()` renamed to `processMailbox()` and simplified to read the slot and call `handleEvent()`; no queue drain loop, no deferred intent handling
- [x] `isIntentEvent()` removed (deferred intent no longer exists)
- [x] `EventPolicy` parameter removed from `postEvent()` signature; `EventPolicy` enum + X-macro + `toString()` removed from supervisor.h
- [x] All call sites updated: `postEvent(...)` calls drop the `EventPolicy` argument; `processEventQueue()` calls renamed to `processMailbox()`
- [x] `#include <freertos/queue.h>` removed from supervisor.h; `TickType_t`, `pdTRUE`, `pdFALSE`, `pdMS_TO_TICKS` native stubs removed
- [x] `QueuedEvent` private struct removed
- [x] Thread safety maintained: `event`/`reason` are written before `pending = true`; slot reads pair `pending` check with data read
- [x] `handleEvent()` and `checkTransitionTimeouts()` unchanged
- [x] All previously passing tests remain passing
- [x] Debug and production builds succeed

## Notes
After this story, `processMailbox()` replaces `processEventQueue()`. `EventPolicy` is removed from the codebase entirely — with a single-slot Mailbox, every write succeeds and there is no backpressure to manage.

### Thread safety approach
On ESP32-S3, stores from a single core are observed in program order by other cores through the bus fabric. The write sequence is: (1) write `reason`, (2) write `event`, (3) write `pending = true`. The reader on Core 0 checks `pending` first; if true, it reads `event` and `reason`, then clears `pending`. No mutex is needed for last-write-wins semantics.
