# US-0022: Configurable Volume Step Resolution

> **Status:** Done  
> **Priority:** Medium  
> **Created:** 2026-04-03

---

## User Story

As a **user**,  
I want **volume control to support a configurable number of steps**,  
so that **I can tune between coarse and fine volume behavior without changing command semantics**.

---

## Acceptance Criteria

- [x] The configured volume step count is applied via `IAudioPlayer::setVolumeSteps(uint8_t)` during audio initialization
- [x] `AUDIO_VOLUME_STEPS` in `src/core/config.h` remains the single source of configuration for step resolution
- [x] Allowed range is enforced to match library constraints (21..255)
- [x] Values below minimum are clamped or rejected with deterministic behavior (documented)
- [x] Values above maximum are clamped or rejected with deterministic behavior (documented)
- [x] Existing `volume [0-N]` CLI behavior remains unchanged except for the dynamic upper bound `N`
- [x] CLI output/help reflects the effective volume range after initialization
- [x] Native tests validate step configuration and range handling in command logic (no hardware required)
- [x] Debug and production builds succeed

---

## Notes

- This story is intentionally separated from `US-0007` (mute-first clean stop).
- Stop behavior must continue to use `setMute(true)` + `stopSong()` and must not depend on changing volume steps.
- If the selected library version changes constraints, this story must update the accepted range and related validation.

### Completion Notes

- Audio initialization now applies configured step resolution via `setVolumeSteps(AUDIO_VOLUME_STEPS)` before default volume is set.
- `AUDIO_VOLUME_STEPS` remains centralized in `src/core/config.h` and is protected by `static_assert(AUDIO_VOLUME_STEPS >= 21 && AUDIO_VOLUME_STEPS <= 255)`.
- Deterministic out-of-range behavior is documented and enforced via compile-time rejection for invalid step configuration.
- CLI keeps `volume [0-N]` semantics and reflects the dynamic upper bound in help and user feedback output.
- Native command-logic tests validate max range handling (`255` accepted, `256` rejected).
- Validation run completed successfully with:
	- `~/.platformio/penv/bin/platformio test --environment native`
	- `~/.platformio/penv/bin/platformio run --environment debug`
	- `~/.platformio/penv/bin/platformio run --environment production`
