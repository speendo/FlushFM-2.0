# US-0005: Library Log Alignment

> **Status:** Done
> **Priority:** Medium
> **Created:** 2026-03-11

---

## User Story

As a **developer**,
I want **all external library log output to be routed through our debug macros**,
so that **production builds stay silent for non-critical messages and debug builds expose full library diagnostics without any code change between environments**.

---

## Acceptance Criteria

- [x] `audio_info` callback routes all messages to `DEBUG_LOG` (silent in production, visible in debug)
- [x] All other existing library callbacks (`audio_showstation`, `audio_showstreamtitle`, `audio_bitrate`, etc.) also use `DEBUG_LOG`
- [x] A production build (`[env:production]`) produces zero serial output from library callbacks during normal playback
- [x] A debug build (`[env:debug]`) shows full library diagnostic output including "slow stream" warnings
- [x] `CORE_DEBUG_LEVEL=0` remains set in `[env:production]` to also suppress `log_e()` output from the library (already in place — verify it is not accidentally changed)
- [x] The library integration rule added to `requirements/guidelines/debug.md` (see Notes — already done as part of this story) is applied

---

## Notes

**Library logging analysis (ESP32-audioI2S):**

- The library exposes **one** logging callback: `audio_info(const char* msg)`. There is no separate `audio_error` or `audio_warning` callback — all informational and warning messages (including "slow stream, dropouts are possible") arrive through the same pipe as a plain string.
- Hard errors inside the library use `log_e()` (ESP-IDF native logger), which is **independent of our callback** and controlled by `-DCORE_DEBUG_LEVEL` in `platformio.ini`. Already set to `0` in `[env:production]`, so these are already silent there.
- `-DAUDIO_LOG` (optional compile flag) enables extra verbose library output, still routed through `audio_info`. Do not add this flag to `[env:production]`.
- The `[INFO]` prefix visible in the serial monitor comes from **our** `PROD_LOG` format string, not from the library.

**Severity policy (decided):** Two-tier mapping is sufficient:

| Library mechanism | Our macro |
|---|---|
| `audio_info()` callback (all messages) | `DEBUG_LOG` |
| `log_e()` (ESP-IDF hard errors) | controlled by `CORE_DEBUG_LEVEL` — no change needed |

Message-level filtering inside `audio_info` (e.g. promoting "slow stream" to `PROD_LOG`) is intentionally avoided: the library fires this message repeatedly (every >100 slow buffer fills), which would violate the Throttling rule in `debug.md`. Accept it as debug-only information.

**Scope:** `src/main.cpp` callback functions only; no change to `debug.h` macros required.

## Test Strategy

**Where tests happen:**

- **Static verification in source code:** confirm library callback functions are routed to `DEBUG_LOG` in the callback implementation file.
- **Build verification in existing PlatformIO environments:**
	- `[env:debug]` for debug-visible callback diagnostics
	- `[env:production]` for silent callback output in production
- **Runtime verification on hardware (manual):** serial monitor checks during real playback in both environments.

**What is explicitly not covered by native unit tests:**

- This story targets library callback routing and environment-dependent logging behavior, which depends on Arduino runtime + ESP32-audioI2S integration.
- Existing native unit tests (e.g. `test/test_cli_command_logic/test_main.cpp`) remain useful for command logic regression but are not the primary verification mechanism for US-0005.

**Manual runtime checklist:**

1. Flash and run `[env:debug]`, start playback, verify callback diagnostics are visible (including slow-stream warnings when they occur).
2. Flash and run `[env:production]`, start normal playback, verify zero serial output from library callback paths.
3. Confirm `-DCORE_DEBUG_LEVEL=0` remains present in `[env:production]`.
