# US-0012: Documentation Refresh and User Story Status System

> **Status:** Done  
> **Priority:** Medium  
> **Created:** 2026-03-28

---

## User Story

As a **developer and maintainer**,  
I want **project documentation (especially README.md) to be updated and user story status tracking to be standardized**,  
so that **the current implementation state is transparent and open vs. completed work is easy to identify**.

---

## Acceptance Criteria

- [x] `README.md` is updated to reflect the current implemented feature set and current development workflow
- [x] Documentation in `docs/` is reviewed and updated where needed for consistency with current behavior
- [x] `US-0008` is refined to require reliable LED mapping discovery and mandatory post-implementation documentation updates
- [x] A simple mechanism to distinguish open vs. completed stories is introduced and documented using folder-based status organization

---

## Investigation Notes

- Current story files contain status fields, but status usage is not consistently optimized for quick project-level overview.
- README and documentation have evolved over multiple stories and should be synchronized with the latest architecture and implemented features.
- LED behavior and mapping are currently not documented clearly enough for hardware bring-up and troubleshooting.
- A lightweight folder-based status structure can reduce friction during planning and reviews.

---

## Suggested Implementation Options

1. Move stories into `requirements/user-stories/open/`, `requirements/user-stories/in-progress/`, and `requirements/user-stories/done/`
2. Keep status in each story front section and enforce exact wording via guideline
3. Add README section "Project Status" describing folder-based status tracking
4. Add documentation rule update in `requirements/guidelines/documentation.md` for mandatory story status format and folder consistency
