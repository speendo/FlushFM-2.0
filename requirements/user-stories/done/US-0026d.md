# US-0026d: API Migration (register/status/required)

> **Status:** Done | **Priority:** Medium | **Created:** 2026-04-25

## User Story
As a **maintainer**, I want **to move key controller APIs to identifier-based logic**, so that **name-based lookup is removed from critical paths**.

## Acceptance Criteria
- [x] Name-based lookup is removed from migrated APIs
- [x] Deterministic direct access path is used
- [x] Graceful degradation remains unchanged for optional components

## Notes
This story depends on US-0026c and focuses on API layer migration: registerComponent, getComponentStatus, isComponentRequired.

### Implementation
- `componentRegistry_` changed from `std::map<std::string, ComponentRegistryEntry>` to `std::array<ComponentRegistryEntry, static_cast<size_t>(ComponentID::Count)>`
- `ComponentRegistryEntry` gained `isRegistered` bool to track which components are actually registered (array always has all slots)
- `componentIdFromName()` helper added to anonymous namespace — linear scan of 4 component names, returns `ComponentID::Count` for unknown names
- All 3 `ComponentID` overloads use direct array access (no string delegation)
- All 3 `const char*` overloads use `componentIdFromName()` + array access
- Collateral: `setComponentTransitionHooks`, `markComponentFailed`, `beginComponentTransition`, `reportCompletion`, `beginOrchestration` all updated from map find/emplace to array access
- `beginOrchestration` now checks `entry.isRegistered` to only create pending transitions for registered components
- `#include <array>` added to supervisoh. `#include <map>` stays (needed by pendingTransitions_ and componentHooks_)
