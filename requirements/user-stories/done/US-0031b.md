# US-0031b: ObservedState Advances Only After All Components Committed

> **Status:** Done | **Priority:** High | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **`observedState_` to lag behind `targetMode_` and advance only when all components confirm COMMITTED**, so that **observers see confirmed reality, not aspirational intent, and the Supervisor follows the components per state-management.md**.

## Acceptance Criteria
- [x] `targetMode_` is set when a transition is requested (via `requestStateTransition` or direct `transitionTo`)
- [x] `observedState_` does NOT change during `transitionTo()` — it stays at the previous state
- [x] `observedState_` advances to `targetMode_` only inside `reportCompletion()` after all pending transitions resolve and `requiredFailure` is false
- [x] Direct transitions (no components involved: BOOT→SLEEP, WIFI_DISCONNECTED→ERROR, etc.) advance `observedState_` immediately since no orchestration is needed
- [x] `state()` continues to return `observedState_` (observers see confirmed state)
- [x] State observers (registered via `subscribe()`) still receive `observedState_` on state change — fire only when `observedState_` actually changes
- [x] `beginOrchestration()` stores the step target in `orchestration_.target`. `targetMode_` is the ultimate goal set by `handleEvent()` user intents — it is NOT overwritten by `beginOrchestration()`
- [x] `transitionTo()` becomes an internal commit helper: applies `observedState_ = next`, logs transition, notifies observers. Called only from reportCompletion or direct-event paths.
- [x] `handleEvent()` no longer calls `transitionTo()` directly for orchestrator-driven transitions (PLAY_REQUESTED, STOP_REQUESTED, ENTER_SLEEP). Instead it sets `targetMode_` and starts orchestration. The actual state change happens in `reportCompletion`.
- [x] All 64 native tests pass with updated assertions where needed (some tests may need to verify `observedState_` vs `targetMode_` distinction)
- [x] Debug and production builds succeed

## Notes

### Core semantic change

**Before:**
```cpp
// handleEvent for PLAY_REQUESTED/SLEEP
transitionTo(SystemState::CONNECTING, event, reason);  // state changes NOW
if (bothReady) requestStateTransition(SystemState::READY);
```

**After:**
```cpp
// handleEvent for PLAY_REQUESTED/SLEEP
targetMode_ = SystemState::CONNECTING;
if (bothReady) requestStateTransition(SystemState::READY);  // sets targetMode_ = READY
// state() still returns SLEEP until orchestration completes
```

### Two categories of state change

1. **Direct transitions** — no components involved, `observedState_` advances immediately:
   - BOOTING + BOOT → SLEEP
   - CONNECTING/READY/LIVE + WIFI_DISCONNECTED → ERROR
   - READY + COMPONENT_SETUP_FAILED → ERROR
   - CONNECTING/LIVE + AUDIO_INIT_FAILED → ERROR
   - ERROR + RECOVER → READY (until US-0031c changes this to BOOTING)

2. **Orchestrated transitions** — `observedState_` advances only when all components COMMITTED:
   - CONNECTING → READY (via WIFI_READY + AUDIO_INIT_OK)
   - READY → LIVE (via PLAY_REQUESTED / deferred target check)
   - LIVE → READY (via STOP_REQUESTED or replay)
   - LIVE → SLEEP (via ENTER_SLEEP)
   - ERROR → SLEEP (via ENTER_SLEEP)

### transitionTo() changes

`transitionTo()` becomes a private commit function called only when the state change is confirmed:

```cpp
void Supervisor::transitionTo(SystemState next, SystemEvent trigger, SystemReason reason, uint32_t transitionId) {
    if (next == observedState_) return;
    const SystemState previous = observedState_;
    observedState_ = next;
    // error/targetMode clearing logic unchanged
    PROD_LOG(...);
    for (observer) observer(next);  // observers see confirmed state
    // deferred replay logic removed (US-0025g already handled this)
}
```

The `targetMode_` clearing logic (`if (next == targetMode_) targetMode_ = SLEEP`) stays.

### handleEvent() changes

All `transitionTo()` calls for user intents change to setting `targetMode_` + `requestStateTransition()`:

| Event | Old | New |
|-------|-----|-----|
| ENTER_SLEEP | `requestStateTransition(SLEEP)` | `targetMode_ = SLEEP; requestStateTransition(SLEEP)` |
| STOP_REQUESTED | `requestStateTransition(READY)` | `targetMode_ = SLEEP; requestStateTransition(READY)` |
| PLAY_REQUESTED/SLEEP | `transitionTo(CONNECTING)` + conditional `requestStateTransition(READY)` | `targetMode_ = CONNECTING` + conditional `requestStateTransition(READY)` |
| PLAY_REQUESTED/LIVE | `requestStateTransition(READY)` | `targetMode_ = LIVE; requestStateTransition(READY)` |
| PLAY_REQUESTED/other | `requestStateTransition(LIVE)` | `targetMode_ = LIVE; requestStateTransition(LIVE)` |

(Note: `targetMode_ = LIVE` in the LIVE replay case is already done by US-0025g.)

Direct event handlers (BOOT→SLEEP, WIFI_DISCONNECTED→ERROR, etc.) still call `transitionTo()` directly — these have no components and should advance `observedState_` immediately.

### beginOrchestration() — no change needed

`beginOrchestration()` does NOT set `targetMode_`. The step target is stored in `orchestration_.target` (already the case). `targetMode_` is the ultimate goal set by `handleEvent()` user intents and must not be overwritten by intermediate orchestration steps — the deferred play check (`targetMode_ == LIVE` in `reportCompletion`) depends on this.

### reportCompletion() — no change needed

`reportCompletion()` already calls `transitionTo()` after all pending transitions resolve. This advances `observedState_` to the orchestration target at the correct time (after component confirmation). No code change required.

### ObservedState advancement summary

| Transition type | Called from | observedState_ advances |
|----------------|-------------|------------------------|
| Direct (BOOT→SLEEP, ERROR paths) | `handleEvent()` | Immediately (in `transitionTo()`) |
| Orchestrated (CONNECTING→READY, etc.) | `reportCompletion()` | After all components COMMITTED |

### Test impact

No existing tests break — the lag semantics were already tested implicitly by existing state-machine tests (e.g., `test_play_requested_during_connecting_is_deferred_until_ready` checks `state() == CONNECTING` during orchestration). One new explicit test (`test_observed_state_lags_until_orchestration_confirms`) verifies the lag with clear assertions.

All 65 tests pass (64 old + 1 new), 4 pre-existing errors unchanged.
