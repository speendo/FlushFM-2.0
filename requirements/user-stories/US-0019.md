# US-0019: Component Architecture – Producer-Side Event Deduplication (Optional)

> **Status:** To Do  
> **Priority:** Low  
> **Created:** 2026-03-29

---

## User Story

As a **light sensor component** (or any sensor posting frequent events),  
I want **to only post an event if the sensed value actually changed**,  
so that **the event queue doesn't flood with redundant ON/ON/ON or OFF/OFF/OFF messages**.

---

## Acceptance Criteria

- [ ] LightSensor component stores lastReportedState internally
- [ ] Before posting event: compare current sensed value to lastReportedState
- [ ] Only post event if state genuinely changed (not just ADC noise or rounding)
- [ ] Optional: implement hysteresis or moving average on raw ADC value to reduce noise
- [ ] Same deduplication pattern applied to Volume/Potentiometer events
- [ ] Test: rapid light level fluctuations at threshold generate only one transition event in queue
- [ ] No change to queue architecture (still single FIFO); producer handles dedup
- [ ] Sensor components document their deduplication threshold if applicable

---

## Notes

This is a low-priority optimization. If queue performance is sufficient in practice, this can be deferred.
The main goal is preventing sensor noise from flooding the queue with identical events.
No controller-side coalescing; producer-side is simpler and more efficient.
