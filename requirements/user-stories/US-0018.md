# US-0018: Component Architecture – Unified Logging for State Transitions

> **Status:** To Do  
> **Priority:** Medium  
> **Created:** 2026-03-29

---

## User Story

As a **developer debugging state transitions**,  
I want **consistent log messages showing which component did what during state changes**,  
so that **I can trace state transitions easily without guessing where failures occurred**.

---

## Acceptance Criteria

- [ ] Log message format defined: `[TransitionId] Component <name>: <TransitionStatus> (<debugReason if Failed>)`
- [ ] Logs emitted at key points: start of component.setXXX(), each reportCompletion(), timeout event, final commit
- [ ] Example log trace for "OFF → IDLE" transition visible in serial monitor
- [ ] Debug text (debugReason) is static const char* (never dynamic strings)
- [ ] Successful completions require no extra debug output in normal flow (clean logs)
- [ ] Failed transitions always log debug reason (no silent failures)
- [ ] Use existing DEBUG_LOG, PROD_LOG, ERROR_LOG macros (no new logging framework)
- [ ] transitionId is printed in all logs for correlation
- [ ] Test: capture full log of "OFF → IDLE → STREAMING → IDLE → OFF" and verify logs are complete and readable

---

## Notes

Logs should tell the story of what happened during transition without being verbose.
Keep output readable in serial monitor; avoid log spam.
No timestamp duplication (ESP32 framework already adds millis-like markers).
