# US-0004b: Component Lifecycle Integration Contract

> **Status:** Done  
> **Priority:** High  
> **Created:** 2026-03-28

---

## User Story

As a **developer**,  
I want **each component to integrate through a clear lifecycle contract**,  
so that **setup and runtime responsibilities are consistent across modules**.

---

## Acceptance Criteria

- [x] A common component lifecycle contract is introduced for orchestration-level integration
- [x] Every component participating in orchestration exposes setup behavior through the contract
- [x] Loop behavior is supported but optional for event-driven components
- [x] `main.cpp` integrates components through the lifecycle contract instead of ad-hoc calls
- [x] `main.cpp` uses only contract-level lifecycle entry points and does not call module-specific startup methods directly
- [x] Event-driven modules are not forced into artificial polling loops
- [x] Component dependencies are injected and not created internally
- [x] No direct component-to-component state mutation is introduced

---

## Notes

- `setup` is mandatory for integrated components.
- `loop` is optional by design to keep WiFi and CLI efficient.
- Naming consistency is enforced at orchestration level (`setup`, optional `loop`); module-internal method names (`init`, `start`) are allowed behind the contract.
- Verified with `platformio run --environment debug` (SUCCESS).
- Suggested implementation sequence:
	1. Define a minimal lifecycle contract interface (`setup`, optional `loop`).
	2. Adapt existing components to the contract without changing runtime behavior.
	3. Update orchestration to use contract-based registration and dispatch.
