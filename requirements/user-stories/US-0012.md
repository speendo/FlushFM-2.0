# US-0012: Documentation Refresh and User Story Status System

> **Status:** To Do  
> **Priority:** Medium  
> **Created:** 2026-03-28

---

## User Story

As a **developer and maintainer**,  
I want **project documentation (especially README.md) to be updated and user story status tracking to be standardized**,  
so that **the current implementation state is transparent and open vs. completed work is easy to identify**.

---

## Acceptance Criteria

- [ ] `README.md` is updated to reflect the current implemented feature set and current development workflow
- [ ] Documentation in `docs/` is reviewed and updated where needed for consistency with current behavior
- [ ] Hardware documentation includes LED pin assignment and electrical usage notes
- [ ] A simple mechanism to distinguish open vs. completed stories is introduced and documented

---

## Investigation Notes

- Current story files contain status fields, but status usage is not consistently optimized for quick project-level overview.
- README and documentation have evolved over multiple stories and should be synchronized with the latest architecture and implemented features.
- LED behavior and mapping are currently not documented clearly enough for hardware bring-up and troubleshooting.
- A lightweight status index (or status sections in a central document) can reduce friction during planning and reviews.

---

## Suggested Implementation Options

1. Add a `requirements/user-stories/INDEX.md` grouped by status (`To Do`, `In Progress`, `Done`)
2. Keep status in each story front section and enforce exact wording via guideline
3. Add README section "Project Status" linking to stories by status
4. Add documentation rule update in `requirements/guidelines/documentation.md` for mandatory story status format
