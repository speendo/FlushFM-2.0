# US-0004c: Explicit System State Machine and Transition Control

> **Status:** To Do  
> **Priority:** High  
> **Created:** 2026-03-28

---

## User Story

As a **developer**,  
I want **an explicit system state machine with centralized transitions**,  
so that **runtime behavior is deterministic and aligned with architecture guidelines**.

---

## Acceptance Criteria

- [ ] A `SystemState` model is implemented with `OFF`, `STARTING`, `IDLE`, `STREAMING`, and `ERROR`
- [ ] A single `SystemController` on Core 0 is introduced as the only source of truth for system state
- [ ] State transitions are event-driven and explicitly defined in one place
- [ ] Components can observe state changes through callback registration
- [ ] Core 1 does not write system state directly and signals cross-core events via queue
- [ ] Entering `OFF`, `STARTING`, or `IDLE` clears transient error flags for clean recovery
- [ ] Transition logs include source state, target state, trigger event, and reason

---

## Notes

- Boot-time hardware initialization may bypass formal transitions before scheduler start, per guideline exception.
- This story defines the transition framework, not full feature logic for all future events.
- Suggested implementation sequence:
	1. Introduce `SystemState` and `SystemController` on Core 0.
	2. Define explicit transition events and transition guards in one central path.
	3. Add observer callbacks and Core 1 -> Core 0 queue signaling for state-relevant events.
