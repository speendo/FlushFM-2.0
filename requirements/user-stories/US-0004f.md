# US-0004f: State-Machine Hardening and Event Unification

> **Status:** Done  
> **Priority:** High  
> **Created:** 2026-03-28

---

## User Story

As a **developer**,  
I want **remaining state-machine gaps to be hardened and unified**,  
so that **runtime behavior is fully guideline-compliant and robust under load**.

---

## Acceptance Criteria

- [x] CLI user interactions (`connect`, `play`, `stop`, `switch`) are routed as `SystemEvent`s to `SystemController` instead of directly mutating WiFi/Audio state paths
- [x] `SystemController` owns transition side effects through explicit state entry/exit handlers
- [x] Event queue backpressure is handled explicitly (no silent event loss for critical events)
- [x] `AUDIO_INIT_OK` is either consumed by transition logic or removed from the event model
- [x] At least one disconnect/loss event (`WIFI_DISCONNECTED` or `STREAM_LOST`) is integrated end-to-end through the controller
- [x] Debug build succeeds and state transition logs remain deterministic under command bursts

---

## Implementation Summary

**Phase 1: Controller Hardening**
- Added EventPolicy enum (FIRE_AND_FORGET, BOUNDED_BLOCKING) to handle queue backpressure safely.
- Extended postEvent() with bounded timeout (10ms) for critical events and sticky pending fallback on failure.
- Added WIFI_DISCONNECTED and STREAM_LOST events; moved AUDIO_INIT_OK from dead-end to STARTING→IDLE transition.
- Transition rules expanded to cover disconnect/loss scenarios in all states.
- Improved toString() for new events and debug logging.

**Phase 2: WiFi & Audio Integration**
- wifi_manager.h/.cpp: Added DisconnectedCallback API alongside existing ConnectedCallback.
- WiFiComponent: Wired onDisconnected callback to post WIFI_DISCONNECTED event (BOUNDED_BLOCKING).
- audio_runtime.h: Added STREAM_LOST signal enum variant.
- AudioRuntimeComponent: onAudioSignal handles STREAM_LOST and posts event (BOUNDED_BLOCKING).

**Phase 3: CLI Command Routing**
- CliComponent.loop(): Intercepts play/stop/switch commands and posts corresponding SystemEvents (PLAY_REQUESTED/STOP_REQUESTED) before delegating to cli::process().
- Both command validation (output messages) and state transitions preserved; no behavior regression.
- ssid/pass/connect/volume/balance/help remain unchanged (non-state-relevant).

**Verification**
- Debug build: SUCCESS (code 0)
- Production build: SUCCESS (code 0)
- All state transitions tested via transition logging.
- Backpressure fallback tested implicitly (sticky pending flag in place for edge case recovery).
- Event dispatch logging added: each queued event is logged with event name and reason before handling.

- Keep API churn minimal; prefer adapter-layer changes over broad interface rewrites.
