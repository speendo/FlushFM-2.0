# US-0004f: State-Machine Hardening and Event Unification

> **Status:** To Do  
> **Priority:** High  
> **Created:** 2026-03-28

---

## User Story

As a **developer**,  
I want **remaining state-machine gaps to be hardened and unified**,  
so that **runtime behavior is fully guideline-compliant and robust under load**.

---

## Acceptance Criteria

- [ ] CLI user interactions (`connect`, `play`, `stop`, `switch`) are routed as `SystemEvent`s to `SystemController` instead of directly mutating WiFi/Audio state paths
- [ ] `SystemController` owns transition side effects through explicit state entry/exit handlers
- [ ] Event queue backpressure is handled explicitly (no silent event loss for critical events)
- [ ] `AUDIO_INIT_OK` is either consumed by transition logic or removed from the event model
- [ ] At least one disconnect/loss event (`WIFI_DISCONNECTED` or `STREAM_LOST`) is integrated end-to-end through the controller
- [ ] Debug build succeeds and state transition logs remain deterministic under command bursts

---

## Notes

- Scope intentionally excludes UI/relay feature expansion; focus is state-path correctness and robustness.
- Keep API churn minimal; prefer adapter-layer changes over broad interface rewrites.
- Suggested implementation sequence:
  1. Add CLI -> controller event adapter for command-triggered state changes.
  2. Centralize transition side effects in controller entry/exit hooks.
  3. Harden queue semantics and close remaining dead event paths.
