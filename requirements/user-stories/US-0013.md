## Test Strategy

**Write tests first (TDD per `testing.md`):**
- [ ] Unit test: `TransitionStatus` enum has exactly two values (`Completed`, `Failed`)
- [ ] Unit test: `ComponentLifecycleStatus` enum has expected values
- [ ] Unit test: `DebugReason` constants are correctly defined as static `const char*`
- [ ] Unit test: `toString()` helper functions return correct string representations for all enum values
- [ ] Unit test: `toString()` handles invalid enum values gracefully (return "UNKNOWN" or error)
- [ ] Unit test: No null pointer dereference when passing `DebugReason = nullptr` to log functions
- [ ] Corner case: Empty string `DebugReason` is handled safely (no buffer overflow)
- [ ] Corner case: Maximum length `DebugReason` (e.g., 256 characters) is stored/logged without truncation issues
- [ ] Unit test: All `DebugReason` constants are non-null and point to valid memory

**Runtime Safety:**
- [ ] No memory leaks when repeatedly constructing/destroying enum values
- [ ] No stack overflow from recursive `toString()` calls
- [ ] Bounds check: enum values cannot be cast from invalid integers (e.g., -1, 999)

Tests run on host via `[env:native]` before implementation.

Keep types simple and MCU-friendly: static strings only (const char*), no dynamic allocation.
