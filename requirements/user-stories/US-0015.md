# US-0015: Component Architecture – Component Lifecycle Registry in SystemController

> **Status:** To Do  
> **Priority:** High  
> **Created:** 2026-03-29

---

## User Story

As a **system controller**,  
I want **to track which components are present, required, and healthy**,  
so that **I can make correct decisions about required vs optional failures during state transitions**.

---

## Acceptance Criteria

- [ ] SystemController stores per-component: {name, lifeCycleStatus, isRequired, isDisabled, lastFailureReason}
- [ ] Add method: `registerComponent(const char* name, bool isRequired)` with logging
- [ ] Add method: `getComponentStatus(const char* name) → ComponentLifecycleStatus`
- [ ] Add method: `markComponentFailed(const char* name, const char* reason)` with logging
- [ ] Add method: `isComponentRequired(const char* name) → bool`
- [ ] Required/optional is defined once per component in controller-owned configuration (not per target state)
- [ ] Registry persists only for one boot (resets to Unknown on startup)
- [ ] No hardcoding of component lists; registry is data-driven
- [ ] Components can opt-in to register themselves
- [ ] All existing components register themselves on startup
- [ ] No compilation errors

---

## Test Strategy

**Write tests first (TDD per `testing.md`):**
- [ ] Unit test: `registerComponent(name, isRequired)` stores registry entry and returns success
- [ ] Unit test: `getComponentStatus(name)` returns correct status after registration
- [ ] Unit test: `markComponentFailed(name, reason)` updates component state and logs failure
- [ ] Unit test: `isComponentRequired(name)` returns correct required/optional classification
- [ ] Unit test: Registry handles duplicate registration gracefully (skip or update)
- [ ] Unit test: Unknown component queries return sensible default (Unknown status, required=false)
- [ ] Unit test: Registry resets on boot (new SystemController instance has empty registry)

**Corner Cases & Runtime Safety:**
- [ ] Unit test: `registerComponent(nullptr, true)` rejected or handled safely (no crash)
- [ ] Unit test: `registerComponent("", true)` (empty string) rejected or handled safely
- [ ] Unit test: `registerComponent()` with component name > 256 chars does not overflow buffer
- [ ] Unit test: `markComponentFailed(valid_name, nullptr)` accepted (reason is optional)
- [ ] Unit test: `markComponentFailed(valid_name, long_reason)` with reason > 512 chars does not overflow
- [ ] Unit test: Registering 100+ components does not crash map or cause memory issues
- [ ] Unit test: Registry iteration is safe during concurrent modifications (if multi-threaded)
- [ ] No use-after-free: component names stored as copies (not pointers to caller's stack)
- [ ] No memory leaks when clearing/destroying registry
- [ ] No null pointer dereference in `getComponentStatus()`, `isComponentRequired()` edge cases

Tests use mock components; no real hardware needed. Run on host via `[env:native]`.

---

## Notes

Registry uses std::map<const char*, ComponentRegistryEntry> internally.
Components register during setup or orchestration can register them manually.
This is the foundation for required/optional policy in transitions.
