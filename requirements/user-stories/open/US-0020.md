# US-0020: Component Architecture – Unit and Integration Tests for New Orchestration

> **Status:** To Do  
> **Priority:** High  
> **Created:** 2026-03-29

---

## User Story

As a **QA engineer**,  
I want **to verify that required/optional policy, timeouts, and failed transitions work as designed**,  
so that **the system behavior is predictable, testable, and regression-free**.

---

## Acceptance Criteria

- [ ] Unit test: Mock component returns timeoutMs and reports `Completed` callback immediately
- [ ] Unit test: Mock component returns timeoutMs and reports `Completed` callback later (async)
- [ ] Unit test: Mock component returns timeoutMs and reports `Failed` callback (required) → transition should NOT commit
- [ ] Unit test: Mock component returns timeoutMs and reports `Failed` callback (optional) → transition should commit with "degraded" flag
- [ ] Unit test: Component timeout expires (no callback within timeout) → SystemController calls `onTransitionTimeout(transitionId)`
- [ ] Unit test: Timeout handler transitions component to safe state and emits `Failed(timeout)` callback
- [ ] Unit test: Late callback after timeout is ignored
- [ ] Unit test: Stale callback with old transitionId is ignored
- [ ] Integration test: OFF → IDLE with all required components successfully completing
- [ ] Integration test: OFF → IDLE with one optional component failing → overall Degraded but IDLE reached
- [ ] Integration test: OFF → IDLE with one required component failing → transition reverted, error state entered
- [ ] Integration test: Verify logging output contains all expected transitionId and component names
- [ ] Integration test: stop/shutdown flow waits for the audio completion callback before the controller continues to later power-off steps
- [ ] All tests pass in native environment (`platformio test --environment native`)
- [ ] Test framework uses mock ISystemComponent instances, no real hardware required
- [ ] Coverage: at least required/optional boolean, Completed/Failed callback statuses, timeout scenarios
- [ ] Tests use component-level required/optional classification from SystemController configuration (not per target state)

**Corner Cases & Runtime Error Prevention:**
- [ ] Unit test: `reportCompletion(nullptr, ...)` does not crash (validated early or rejected)
- [ ] Unit test: `setXXX(transitionId=0)` and `setXXX(transitionId=UINT32_MAX)` both handled correctly
- [ ] Unit test: Timeout value = 0 (immediate callback required) handled correctly
- [ ] Unit test: Timeout value = UINT32_MAX does not overflow 32-bit timer
- [ ] Unit test: Rapid successive transitions (> 10 per component/sec) do not cause queue overflow
- [ ] Unit test: Multiple components failing simultaneously (all 4 fail at once) → correct policy applied for each
- [ ] Unit test: Double-callback attempt (same transitionId reported twice) → second is ignored safely
- [ ] Unit test: Race condition: timeout fires while component is reporting `Completed` → last-write-wins or atomic
- [ ] Unit test: Component names > 256 chars do not overflow buffers
- [ ] Unit test: Failure reason strings > 512 chars do not overflow
- [ ] Unit test: Null failure reason accepted (reason is optional)
- [ ] No memory leaks: registry cleared, transitions aborted, callbacks cleaned up
- [ ] No use-after-free: all pointers validated, component names copied (not stored as references)
- [ ] No deadlock: `onTransitionTimeout()` called while `setXXX()` in-flight → no re-entrancy crash
- [ ] No stack overflow: deeply nested callback chains handled safely
- [ ] Graceful restart: if controller crashes mid-transition, no dangling state corrupts next boot

Create test mocks of ISystemComponent in test/ folder.
