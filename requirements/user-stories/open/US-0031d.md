# US-0031d: FATAL Absorbing State

> **Status:** To Do | **Priority:** High | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **FATAL to be an absorbing terminal state reached through the same orchestration flow as all other TargetModes**, so that **the system enters a controlled halt after all components have been given their chance to shut down**.

## Acceptance Criteria
- [ ] FATAL follows the same orchestration contract as all other TargetModes: `targetMode_ = FATAL` → `beginOrchestration(FATAL)` arms all components → each reports `COMMITTED` or `FAILED` → `observedState_` advances to FATAL
- [ ] Once `observedState_ == FATAL`, `processMailbox()` returns immediately — no event processing, no timeout checks
- [ ] `handleEvent()` drops all events when `observedState_ == FATAL` except `SystemEvent::BOOT`
- [ ] `BOOT` event in FATAL calls `transitionTo(BOOTING)` — full system restart
- [ ] Components that cannot reach FATAL (unresponsive) are handled by their FATAL row in the min/max matrix (US-0031e) with a short timeout; Supervisor advances to FATAL regardless of component cooperation
- [ ] `#if !defined(ARDUINO)` test method `triggerFatal()` added — unit-test shortcut that directly sets `observedState_ = FATAL` to verify halt behavior without full orchestration
- [ ] No production code posts FATAL — entry trigger deferred to recovery design (future story series)
- [ ] New test `test_fatal_rejects_all_events_except_boot` in `test_state_transition_flow`
- [ ] All 63 other tests pass unchanged; 4 pre-existing errors unchanged
- [ ] Debug and production builds succeed

## Notes

### Contract: FATAL is a regular TargetMode

FATAL is identical to CONNECTING, READY, or LIVE in flow:

```cpp
// Production entry (future recovery design):
targetMode_ = SystemState::FATAL;
beginOrchestration(SystemState::FATAL, trigger, reason, transitionId);
// → components armed → report COMMITTED/FAILED → observedState_ = FATAL
```

The min/max matrix (US-0031e) defines each component's FATAL row. Per-component FATAL timeouts should be short (e.g., 100ms) — just enough to acknowledge the halt, not do complex teardown. If a component doesn't respond, its timeout fires, the Supervisor treats it as FAILED, and orchestration still completes. The Supervisor always reaches FATAL regardless of component cooperation.

### processMailbox() gate

Engages only at `observedState_ == FATAL` — after orchestration confirmation:

```cpp
void Supervisor::processMailbox() {
    if (observedState_ == SystemState::FATAL) {
        return;
    }
    // ...existing Mailbox drain + timeout checks...
}
```

### handleEvent() gate

```cpp
void Supervisor::handleEvent(SystemEvent event, SystemReason reason) {
    if (observedState_ == SystemState::FATAL) {
        if (event == SystemEvent::BOOT) {
            transitionTo(SystemState::BOOTING, event, reason);
        }
        return;
    }
    // ...existing event handling...
}
```

### triggerFatal() — unit-test shortcut

```cpp
// supervisor.h (public, guarded):
#if !defined(ARDUINO)
    void postEventBuffered(SystemEvent event, SystemReason reason);
    void triggerFatal();
#endif

// supervisor.cpp:
#if !defined(ARDUINO)
void Supervisor::triggerFatal() {
    transitionTo(SystemState::FATAL, SystemEvent::BOOT, SystemReason::NONE);
}
#endif
```

This bypasses orchestration and directly sets `observedState_ = FATAL` so tests can verify the halt gate without requiring a full component orchestration setup. Production entry always goes through `beginOrchestration(FATAL)`.

### Test

Add to `test/test_state_transition_flow/test_main.cpp`:

```cpp
void test_fatal_rejects_all_events_except_boot() {
    Supervisor controller;
    controller.triggerFatal();
    TEST_ASSERT_EQUAL(static_cast<int>(SystemState::FATAL), static_cast<int>(controller.state()));

    // All events are dropped in FATAL
    controller.postEvent(SystemEvent::PLAY_REQUESTED, SystemReason::USER_REQUEST);
    TEST_ASSERT_EQUAL(static_cast<int>(SystemState::FATAL), static_cast<int>(controller.state()));

    controller.postEvent(SystemEvent::STOP_REQUESTED, SystemReason::USER_REQUEST);
    TEST_ASSERT_EQUAL(static_cast<int>(SystemState::FATAL), static_cast<int>(controller.state()));

    controller.postEvent(SystemEvent::WIFI_DISCONNECTED, SystemReason::NONE);
    TEST_ASSERT_EQUAL(static_cast<int>(SystemState::FATAL), static_cast<int>(controller.state()));

    // BOOT recovers from FATAL
    controller.postEvent(SystemEvent::BOOT, SystemReason::BOOT_SEQUENCE);
    TEST_ASSERT_EQUAL(static_cast<int>(SystemState::SLEEP), static_cast<int>(controller.state()));
}
```

Register in main():
```cpp
RUN_TEST(test_fatal_rejects_all_events_except_boot);
```

### FATAL processMailbox guard placement

The guard must appear inside BOTH `#if !defined(ARDUINO)` and `#else` branches of `processMailbox()`:

```cpp
void Supervisor::processMailbox() {
#if !defined(ARDUINO)
    if (observedState_ == SystemState::FATAL) {
        return;
    }
    if (mailbox_.pending) { ... }
    checkTransitionTimeouts();
    return;
#else
    if (observedState_ == SystemState::FATAL) {
        return;
    }
    if (mailbox_.pending) { ... }
    checkTransitionTimeouts();
#endif
}
```

### Why no RetryPolicy?

RetryPolicy (counter, maxRetries, exhaustion) belongs to the error recovery design. Once recovery is designed, the transition into FATAL will be `retryPolicy.isExhausted()` → `targetMode_ = FATAL` → `beginOrchestration(FATAL)`. For now, `triggerFatal()` is the test-only entry point.
