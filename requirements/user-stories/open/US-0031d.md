# US-0031d: FATAL State and Retry Exhaustion

> **Status:** To Do | **Priority:** High | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **the FATAL target mode to halt the software when recovery retries are exhausted**, so that **the system has a defined inert endpoint that requires user intervention to clear**.

## Acceptance Criteria
- [ ] `RetryPolicy` struct tracks recovery attempts: `uint8_t recoveryCounter`, `uint8_t maxRetries` (configurable)
- [ ] On ERROR→BOOTING recovery (from US-0031c), increment `recoveryCounter`
- [ ] If `recoveryCounter >= maxRetries`, set `targetMode_ = FATAL`, call `transitionTo(FATAL)`
- [ ] When `observedState_ == FATAL`, Supervisor processing stops: `state() == FATAL`, no events processed
- [ ] FATAL can be cleared by: (a) posting ENTER_SLEEP event → triggers BOOTING, (b) posting BOOT event → triggers BOOTING
- [ ] Clearing FATAL resets `recoveryCounter = 0`
- [ ] `maxRetries` defaults to 3, configurable via `CONFIG_ERROR_RECOVERY_MAX_RETRIES` in config.h
- [ ] `recoveryCounter` is reset on any successful transition to READY or LIVE (not just on FATAL clear)
- [ ] All existing recovery tests pass (no retry exhaustion in normal recovery)
- [ ] New test: `test_recovery_exhaustion_triggers_fatal` — simulate repeated recovery failures
- [ ] Debug and production builds succeed

## Notes

### New members in supervisor.h

```cpp
struct RetryPolicy {
    uint8_t maxRetries = 3;
    uint8_t counter = 0;

    bool isExhausted() const { return counter >= maxRetries; }
    void increment() { counter++; }
    void reset() { counter = 0; }
};
```

Add to Supervisor private section:
```cpp
RetryPolicy retryPolicy_{};
```

### Recovery counter logic

In handleEvent ERROR handler (after US-0031c rewrite):
```cpp
case SystemState::ERROR:
    if (event == SystemEvent::RECOVER) {
        retryPolicy_.increment();
        if (retryPolicy_.isExhausted()) {
            targetMode_ = SystemState::FATAL;
            transitionTo(SystemState::FATAL, event, reason);
            return;
        }
        targetMode_ = SystemState::READY;
        transitionTo(SystemState::BOOTING, event, reason);
        // ...sequential traversal continues
    }
    break;
```

In reportCompletion, after successful transition to READY or LIVE:
```cpp
if (!orchestration_.requiredFailure) {
    transitionTo(orchestration_.target, ...);
    if (observedState_ == SystemState::READY || observedState_ == SystemState::LIVE) {
        retryPolicy_.reset();
    }
    // ...sequential traversal check
}
```

### FATAL clearing

In handleEvent, before the state switch:
```cpp
if (observedState_ == SystemState::FATAL) {
    if (event == SystemEvent::ENTER_SLEEP || event == SystemEvent::BOOT) {
        retryPolicy_.reset();
        targetMode_ = SystemState::SLEEP;
        transitionTo(SystemState::BOOTING, event, reason);
        // sequential traversal: BOOTING → CONNECTING → READY
    }
    return;  // FATAL processes no other events
}
```

### FATAL state in switch

Add a FATAL case to the state switch in handleEvent:
```cpp
case SystemState::FATAL:
    // No events processed in FATAL — handled above
    break;
```

### FATAL handling in processMailbox / main loop

When `state() == FATAL`, the main loop should still call `processMailbox()` but only BOOT/ENTER_SLEEP events have any effect (all others are dropped by the FATAL gate in handleEvent). No change needed to `main.cpp`.
