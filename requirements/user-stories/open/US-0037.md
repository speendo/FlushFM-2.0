# US-0037: Algorithmic Step-Through for STATE_REQUESTED

> **Status:** To Do | **Priority:** Medium | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want the `STATE_REQUESTED` handler to use a hierarchy-driven step-through algorithm instead of per-target branching, so that transition behavior follows state ordering (`stateRank`) and remains robust when state rules evolve.

## Acceptance Criteria

### External API Guard
- [ ] `postEvent(event, reason, target)` rejects both `BOOTING` and `CONNECTING`
- [ ] Rejected targets are logged and return `false`
- [ ] `BOOTING` behavior remains externally reachable only through a dedicated public method (`initializeSystemComponents()`), not through `postEvent(..., target)`

### Core Algorithm
- [ ] `STATE_REQUESTED` handling in `handleEvent()` is refactored from per-target switch logic to hierarchy-based stepping
- [ ] Direction is determined by rank comparison only: move "up" when `targetRank > observedRank`, move "down" when `targetRank < observedRank`
- [ ] No fixed numeric stepping rule is required by contract (no `+/-10` coupling in story requirements)
- [ ] `targetMode_` stores the user intent and drives continuation across intermediate steps

### Special Cases
- [ ] `ERROR` target triggers immediate non-orchestrated state update; if already in `ERROR`, escalate to `FATAL`
- [ ] `FATAL` target triggers immediate non-orchestrated state update and preserves fatal halt behavior
- [ ] `SLEEP` target from L2 follows downward traversal through L2 first (`LIVE -> READY -> SLEEP` or `READY -> SLEEP`)

### BOOTING Setup Reuse
- [ ] Booting setup logic is centralized in a public `initializeSystemComponents()` method
- [ ] `main::setup()` calls `initializeSystemComponents()`
- [ ] Internal BOOTING flow reuses the same method (no duplicated boot/setup logic)

### Naming Clarity
- [ ] `transitionTo()` is renamed to a name that clearly indicates immediate observed-state update semantics (e.g. `setObservedStateImmediate()`)
- [ ] References and docs in code are updated consistently to the new name

### Behavior Fixes
- [ ] READY request fixes current bug by setting `targetMode_ = READY` (not `SLEEP`)
- [ ] Wake/re-entry path no longer skips BOOTING when BOOTING is required by the traversal logic
- [ ] CONNECTING deferral is preserved: LIVE requested during CONNECTING updates intent and continues after in-flight step completion

### Completion Continuation
- [ ] `reportCompletion()` generalizes continuation: when `observedState_ != targetMode_`, request next hierarchical step automatically
- [ ] Existing LIVE replay behavior remains covered by the generalized continuation logic

### Validation
- [ ] Fatal guard at the top of `handleEvent()` continues to block processing in `FATAL`
- [ ] `COMPONENT_SETUP_FAILED` behavior remains unchanged
- [ ] Debug and production builds succeed
- [ ] Native unit tests pass

## Notes
- The state hierarchy is already encoded: `FATAL(0) < ERROR(10) < SLEEP(20) < BOOTING(30) < CONNECTING(40) < READY(50) < LIVE(60)` via `stateRank()`
- L2 linear sequence is `BOOTING -> CONNECTING -> READY -> LIVE`; traversal is based on state ordering (higher/lower), not on hardcoded arithmetic increments in requirements
- `SLEEP` is L1, not in L2; transitions toward sleep from L2 pass through the defined downward path
- `postEvent(..., target)` is the external request path and must reject internal stepping states (`BOOTING`, `CONNECTING`)
- The current `handleEvent()` block for `STATE_REQUESTED` (around lines 440-484 in current `supervisor.cpp`) is the replacement target for this story
