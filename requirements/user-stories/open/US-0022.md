# US-0022: Configurable Volume Step Resolution

> **Status:** To Do  
> **Priority:** Medium  
> **Created:** 2026-04-03

---

## User Story

As a **user**,  
I want **volume control to support a configurable number of steps**,  
so that **I can tune between coarse and fine volume behavior without changing command semantics**.

---

## Acceptance Criteria

- [ ] The configured volume step count is applied via `IAudioPlayer::setVolumeSteps(uint8_t)` during audio initialization
- [ ] `AUDIO_VOLUME_STEPS` in `src/core/config.h` remains the single source of configuration for step resolution
- [ ] Allowed range is enforced to match library constraints (21..255)
- [ ] Values below minimum are clamped or rejected with deterministic behavior (documented)
- [ ] Values above maximum are clamped or rejected with deterministic behavior (documented)
- [ ] Existing `volume [0-N]` CLI behavior remains unchanged except for the dynamic upper bound `N`
- [ ] CLI output/help reflects the effective volume range after initialization
- [ ] Native tests validate step configuration and range handling in command logic (no hardware required)
- [ ] Debug and production builds succeed

---

## Notes

- This story is intentionally separated from `US-0007` (mute-first clean stop).
- Stop behavior must continue to use `setMute(true)` + `stopSong()` and must not depend on changing volume steps.
- If the selected library version changes constraints, this story must update the accepted range and related validation.
