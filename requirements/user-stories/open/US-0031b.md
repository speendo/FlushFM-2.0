# US-0031b: ObservedState Advances Only After All Components Committed

> **Status:** To Do | **Priority:** High | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **`observedState_` to lag behind `targetMode_` and advance only when all components confirm COMMITTED**, so that **observers see confirmed reality, not aspirational intent, and the Supervisor follows the components per state-management.md**.

## Acceptance Criteria
- [ ] `targetMode_` is set when a transition is requested (via `requestStateTransition` or direct `transitionTo`)
- [ ] `observedState_` does NOT change during `transitionTo()` — it stays at the previous state
- [ ] `observedState_` advances to `targetMode_` only inside `reportCompletion()` after all pending transitions resolve and `requiredFailure` is false
- [ ] Direct transitions (no components involved: BOOT→SLEEP, WIFI_DISCONNECTED→ERROR, etc.) advance `observedState_` immediately since no orchestration is needed
- [ ] `state()` continues to return `observedState_` (observers see confirmed state)
- [ ] State observers (registered via `subscribe()`) still receive `observedState_` on state change — fire only when `observedState_` actually changes
- [ ] `beginOrchestration()` sets `targetMode_` to the orchestration target before arming components
- [ ] `transitionTo()` becomes an internal commit helper: applies `observedState_ = next`, logs transition, notifies observers. Called only from reportCompletion or direct-event paths.
- [ ] `handleEvent()` no longer calls `transitionTo()` directly for orchestrator-driven transitions (PLAY_REQUESTED, STOP_REQUESTED, ENTER_SLEEP). Instead it sets `targetMode_` and starts orchestration. The actual state change happens in `reportCompletion`.
- [ ] All 64 native tests pass with updated assertions where needed (some tests may need to verify `observedState_` vs `targetMode_` distinction)
- [ ] Debug and production builds succeed

## Notes

### Core semantic change

**Before:**
```cpp
// handleEvent for PLAY_REQUESTED/SLEEP
transitionTo(SystemState::CONNECTING, event, reason);  // state changes NOW
if (bothReady) requestStateTransition(SystemState::READY);
```

**After:**
```cpp
// handleEvent for PLAY_REQUESTED/SLEEP
targetMode_ = SystemState::CONNECTING;
if (bothReady) requestStateTransition(SystemState::READY);  // sets targetMode_ = READY
// state() still returns SLEEP until orchestration completes
```

### Two categories of state change

1. **Direct transitions** — no components involved, `observedState_` advances immediately:
   - BOOTING + BOOT → SLEEP
   - CONNECTING/READY/LIVE + WIFI_DISCONNECTED → ERROR
   - READY + COMPONENT_SETUP_FAILED → ERROR
   - CONNECTING/LIVE + AUDIO_INIT_FAILED → ERROR
   - ERROR + RECOVER → READY (until US-0031c changes this to BOOTING)

2. **Orchestrated transitions** — `observedState_` advances only when all components COMMITTED:
   - CONNECTING → READY (via WIFI_READY + AUDIO_INIT_OK)
   - READY → LIVE (via PLAY_REQUESTED / deferred target check)
   - LIVE → READY (via STOP_REQUESTED or replay)
   - LIVE → SLEEP (via ENTER_SLEEP)
   - ERROR → SLEEP (via ENTER_SLEEP)

### transitionTo() changes

`transitionTo()` becomes a private commit function called only when the state change is confirmed:

```cpp
void Supervisor::transitionTo(SystemState next, SystemEvent trigger, SystemReason reason, uint32_t transitionId) {
    if (next == observedState_) return;
    const SystemState previous = observedState_;
    observedState_ = next;
    // error/targetMode clearing logic unchanged
    PROD_LOG(...);
    for (observer) observer(next);  // observers see confirmed state
    // deferred replay logic removed (US-0025g already handled this)
}
```

The `targetMode_` clearing logic (`if (next == targetMode_) targetMode_ = SLEEP`) stays.

### handleEvent() changes

All `transitionTo()` calls for user intents change to setting `targetMode_` + `requestStateTransition()`:

| Event | Old | New |
|-------|-----|-----|
| ENTER_SLEEP | `requestStateTransition(SLEEP)` | `targetMode_ = SLEEP; requestStateTransition(SLEEP)` |
| STOP_REQUESTED | `requestStateTransition(READY)` | `targetMode_ = SLEEP; requestStateTransition(READY)` |
| PLAY_REQUESTED/SLEEP | `transitionTo(CONNECTING)` + conditional `requestStateTransition(READY)` | `targetMode_ = CONNECTING` + conditional `requestStateTransition(READY)` |
| PLAY_REQUESTED/LIVE | `requestStateTransition(READY)` | `targetMode_ = LIVE; requestStateTransition(READY)` |
| PLAY_REQUESTED/other | `requestStateTransition(LIVE)` | `targetMode_ = LIVE; requestStateTransition(LIVE)` |

(Note: `targetMode_ = LIVE` in the LIVE replay case is already done by US-0025g.)

Direct event handlers (BOOT→SLEEP, WIFI_DISCONNECTED→ERROR, etc.) still call `transitionTo()` directly — these have no components and should advance `observedState_` immediately.

### beginOrchestration() changes

```cpp
bool Supervisor::beginOrchestration(SystemState target, ...) {
    const TransitionRequestDecision decision = requestTransition(observedState_, target, transitionId);
    if (decision == TransitionRequestDecision::Ignored || decision == TransitionRequestDecision::Queued) {
        return false;
    }
    targetMode_ = target;  // SET THE TARGET
    // ...rest unchanged...
}
```

### reportCompletion() changes

The `transitionTo()` call moves inside the `if (!hasPendingTransitions())` block and advances `observedState_`:

```cpp
if (orchestration_.active && ... && !hasPendingTransitions()) {
    if (orchestration_.requiredFailure) {
        transitionTo(SystemState::ERROR, ...);  // advances observedState_ to ERROR
    } else {
        transitionTo(orchestration_.target, ...);  // advances observedState_ to target
    }
    // ...rest unchanged...
}
```

### Test impact

Tests that expect `state()` to change BEFORE `completeAllActive()` will break — they must be updated to check that `state()` stays at previous value during orchestration and only changes after `completeAllActive()`. Tests for direct transitions (BOOT→SLEEP, ERROR events) are unaffected.

### Potential test assertions to update

In `test_state_transition_flow`:
- `test_play_requested_during_connecting_is_deferred_until_ready` line 90: `state == CONNECTING` during orchestration → stays SLEEP until completeAllActive
- Several tests that check `state()` during active orchestration — must now check `observedState_` stays at pre-orchestration value

New test needed to verify the lag semantics:
```cpp
void test_observed_state_lags_during_orchestration() {
    TransitionHooksFixture fixture;
    fixture.install();
    fixture.controller.postEvent(SystemEvent::BOOT, SystemReason::BOOT_SEQUENCE);
    TEST_ASSERT_EQUAL(SystemState::SLEEP, fixture.controller.state());
    fixture.controller.postEvent(SystemEvent::WIFI_READY, ...);
    fixture.controller.postEvent(SystemEvent::AUDIO_INIT_OK, ...);
    fixture.controller.postEvent(SystemEvent::PLAY_REQUESTED, ...);
    // State should still be SLEEP — orchestration hasn't completed
    TEST_ASSERT_EQUAL(SystemState::SLEEP, fixture.controller.state());
    fixture.completeAllActive();  // completes CONNECTING→READY
    TEST_ASSERT_EQUAL(SystemState::READY, fixture.controller.state());
}
```
