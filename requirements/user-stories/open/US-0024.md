# US-0024: Audio Fade Envelope for Superseded Transitions

> **Status:** To Do | **Priority:** High | **Created:** 2026-04-17

## User Story
As a **listener**, I want **smooth fade-out and fade-in while state transitions are superseded**, so that **rapid light-trigger reversals do not cause audible jumps or glitches**.

## Acceptance Criteria
- [ ] Audio runtime uses a target-based gain envelope (current gain + target gain), not one-shot transition-bound fades
- [ ] Transition `STREAMING -> READY` sets fade target to muted and does not require hard stop before envelope reaches target
- [ ] Transition `READY -> STREAMING` sets fade target to normal playback level and can supersede a running fade-out
- [ ] If a running fade-out is superseded by fade-in (or vice versa), gain changes direction smoothly without discontinuity
- [ ] Completion semantics for stop transitions are explicit: `Completed` is emitted only after fade-out reached target and streaming is actually stopped
- [ ] Superseded transitions are logged as `superseded` (not `failed`)
- [ ] Unit tests cover fade direction reversal during active envelope updates
- [ ] Unit tests verify no double completion emission during supersede/reversal sequences
- [ ] Native unit tests pass
- [ ] Production build succeeds

## Notes
This story intentionally follows orchestration stories `US-0016` and `US-0017` and must reuse their transitionId and completion callback contract.
The fade envelope implementation should remain component-local (AudioRuntime) and must not introduce cross-component state coupling.
