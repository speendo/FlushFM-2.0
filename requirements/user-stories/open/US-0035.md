# US-0035: Remove Component Commands from Supervisor

> **Status:** To Do | **Priority:** High | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **component commands (WIFI_READY, AUDIO_INIT_OK, etc.) removed from the Supervisor and an Error Event flag mechanism added per state-management.md**, so that **the Supervisor only handles global state transitions and supervisor.cpp is lean**.

## Acceptance Criteria
- [ ] Component command handlers (SLEEP, CONNECTING, READY, LIVE case blocks) removed from `handleEvent()`
- [ ] Error Event flag `{pending, reason, source}` added — set by components, checked in `processMailbox()` to trigger ERROR transitions
- [ ] `setErrorEvent()` method on Supervisor accepts only the first error per transition (duplicates ignored)
- [ ] PLAY_REQUESTED/SLEEP quick path (`getComponentStatus()` readiness check) removed — always orchestrate
- [ ] `COMPONENT_SETUP_FAILED` becomes state-independent in `handleEvent()` — triggers ERROR from any state
- [ ] `WIFI_READY`, `AUDIO_INIT_OK`, `AUDIO_INIT_FAILED`, `WIFI_DISCONNECTED`, `STREAM_LOST` removed from `SystemEvent` enum
- [ ] WiFi and Audio components updated to use `setErrorEvent()` instead of `postEvent(WIFI_DISCONNECTED/AUDIO_INIT_FAILED)`
- [ ] WiFi and Audio components stop posting WIFI_READY/AUDIO_INIT_OK — orchestration handles component readiness via COMMITTED
- [ ] `debug_cli.cpp` error trigger updated from `AUDIO_INIT_FAILED` to `COMPONENT_SETUP_FAILED`
- [ ] `test_state_transition_flow` tests rewritten: component-event tests removed, Error Event tests added, all remaining tests pass with updated flow
- [ ] `test_component_types` SystemEvent round-trip updated to 6 events
- [ ] All tests pass; debug and production builds succeed

## Notes

### Violations addressed

Per `state-management.md` §"Event System": *"Component Commands: Route directly to the target component. Do not pass through the Supervisor."*

| Lines removed from handleEvent | What was there | Why removed |
|------|-----|------|
| 508-517 (SLEEP) | Component commands (WIFI_READY, AUDIO_INIT_OK, etc.) setting component registry lifecycle status | Components track their own state; Supervisor shouldn't manipulate component registry outside orchestration |
| 520-535 (CONNECTING) | Component readiness check + `requestStateTransition(READY)` gated on "both ready" | Orchestration handles readiness via COMMITTED/FAILED — no parallel path needed |
| 538-545 (READY) | Error transitions (WIFI_DISCONNECTED → ERROR, etc.) | Replaced by Error Event flag mechanism |
| 548-555 (LIVE) | Same error transitions | Same replacement |
| 482-484 (PLAY_REQUESTED) | Quick path: skip READY if both components ready | Always orchestrate — if components already ready, they report COMMITTED immediately |

### Remaining handleEvent after cleanup

The switch collapses to empty cases for SLEEP, CONNECTING, READY, LIVE. The state-independent section grows by one (`COMPONENT_SETUP_FAILED`). handleEvent shrink: ~44 lines removed (from ~596 to ~552 total in supervisor.cpp).
