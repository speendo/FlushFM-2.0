# US-0025c: Event Contract Freeze

> **Status:** To Do | **Priority:** High | **Created:** 2026-04-25

## User Story
As a **maintainer**, I want **to freeze the minimal controller-facing event contract and naming semantics**, so that **downstream stories can migrate consistently without contract drift**.

## Acceptance Criteria
- [ ] `docs/event-contract-model.md` is created with: (1) Core Invariants (SSOT, component local states), (2) Event Types (STATE_REQUESTED / STATE_ENTERED / STATE_FAILED semantics), (3) No Intent vs Domain distinction decision, (4) Queue Processing Model (processEventQueue naming, failure handling), (5) Naming Standards table, (6) SystemReason static text contract, (7) Readiness Model (single source), (8) Event Ordering & Arbitration (deterministic, no priority table)
- [ ] `docs/system-controller-api.md` is created with: (1) Module vs Component distinction, (2) Public API surface (state inspection, event posting, queue processing, observer registration, component orchestration), (3) Internal-only methods list (transitionTo, context inspection, diagnostics), (4) Component lifecycle registry ownership, (5) Hook wiring static contract, (6) Graceful degradation guarantees
- [ ] SSOT wording is fixed and unambiguous: "SystemController is the exclusive owner of global system state; components own local states that mirror global guarantees"
- [ ] Queue failure minimum reaction is defined and documented: "On enqueue failure in postEvent(), ERROR_LOG is emitted unconditionally with event context; harder escalation deferred"
- [ ] One canonical naming set is defined: `processEventQueue()` (vs `dispatchPending()`), `STATE_REQUESTED` (vs specific events), `EventPolicy` with `BestEffort`/`Critical`
- [ ] Per-state/per-event outcome matrix template is defined (SystemState × SystemEvent → outcome)
- [ ] State Guarantee Model table is fixed and documented (BOOTING, SLEEP, CONNECTING, READY, LIVE, ERROR with component guarantees)
- [ ] Story-level review checklist is created and approved (verify SSOT consistency, check mirrored state logic, validate naming across code pathways)
- [ ] Open decisions are closed or explicitly converted into follow-up story references (e.g., "harder escalation → future story on ERROR state policy")

## Notes
This story is contract-definition and documentation-freeze only. No code-behavior changes are expected; only documentation, decision recording, and review approval.

Completion means:
1. Both `docs/event-contract-model.md` and `docs/system-controller-api.md` are reviewed and approved
2. Naming standards are agreed upon (no aliases, no migration during implementation)
3. All AC items are verified via inspection/checklist, not code execution
