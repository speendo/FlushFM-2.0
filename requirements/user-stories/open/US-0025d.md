# US-0025d: Enum/String Single Source Mapping

> **Status:** To Do | **Priority:** High | **Created:** 2026-04-25

## User Story
As a **maintainer**, I want **to remove duplicated enum-label maintenance**, so that **enum names and string labels are derived from one central source**.

## Acceptance Criteria
- [ ] A single shared enum-to-string mapping mechanism is defined and approved (e.g., X-macro list, or central mapping table in a dedicated header)
- [ ] The following enums are migrated to use the shared mechanism: (1) `SystemEvent`, (2) `SystemState`, (3) `EventPolicy`, (4) any other enums used in state/event handling
- [ ] For each enum, a corresponding string-label function is generated/derived from the shared definition (e.g., `toString(SystemEvent)` derives from the same source as the enum itself)
- [ ] No duplicate switch-based label tables remain in the codebase; all previous manual `toString()` implementations are removed and replaced by the shared mechanism
- [ ] The shared mechanism is easy to extend: adding a new enum value requires editing only the shared definition, not multiple switch blocks
- [ ] Debug and CLI output uses the shared mechanism for all enum labels (consistency across logging, CLI commands, diagnostic output)
- [ ] All references to enum labels in comments, CLI help text, and documentation reference the shared mechanism (no hardcoded strings)
- [ ] Unit tests verify enum-to-string mapping integrity: (1) round-trip conversion (enum → string → enum) for all values, (2) no gaps or duplicates, (3) label consistency across all output pathways
- [ ] Debug build succeeds
- [ ] Production build succeeds
- [ ] No code-behavior changes; only refactoring of maintenance mechanism

## Notes
This story depends on contract decisions from US-0025c (event types, naming standards). It applies those decisions to the shared enum-to-string mapping.

Scope is maintenance reduction only; no new enum values introduced, no new control-flow paths added. The behavior of the system remains unchanged; only how enum names are maintained is simplified.

Verification:
- Search codebase for `toString(...)` implementations and switch-based label tables; verify all are removed after migration
- Build and run native unit tests to verify mapping integrity
- Spot-check debug output (Serial CLI, logs) for label consistency across different output points
