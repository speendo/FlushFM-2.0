# US-0031: Implement TargetMode and ObservedState Separation

> **Status:** To Do | **Priority:** High | **Created:** 2026-05-08

## User Story
As a **maintainer**, I want **global state split into TargetMode (goal) and ObservedState (current reality) as distinct variables**, so that **the system clearly tracks intent separately from achieved state and can handle transient mismatches**.

## Acceptance Criteria
- [ ] `SystemState` enum is reviewed and mapped to `TargetMode` per guidelines (BOOTING, CONNECTING, READY, LIVE, SLEEP, ERROR, FATAL)
- [ ] A new `TargetMode` enum type is defined (or `SystemState` is renamed if fully migrating)
- [ ] A separate `ObservedState` variable tracks current achieved state in the Supervisor
- [ ] `TargetMode targetMode_` is introduced as the goal/intent state
- [ ] `SystemState observedState_` (or similar) tracks the current state returned by `state()` method
- [ ] Supervisor follows strict rule: `ObservedState` advances only after all components report `COMMITTED`
- [ ] Transition logic uses `TargetMode` to drive orchestration while `ObservedState` lags behind until confirmed
- [ ] State observers receive `ObservedState` notifications, not `TargetMode`
- [ ] Component min/max state matrices reference `TargetMode` dynamically where needed (e.g., backward transitions)
- [ ] Recovery logic reads component current states and calculates highest justified `TargetMode` (not restoring previous)
- [ ] Debug and production builds succeed without errors
- [ ] All native unit tests pass and behavior is unchanged from user perspective (`pio test -e native`)

## Notes
- This is a foundational architectural change aligned with state-management.md guidelines
- Current implementation uses single `state_` variable; refactoring requires careful state-machine verification
- Component tests must validate that transitions still work correctly with separated intent/reality tracking
- This story may require several iterations to handle all edge cases (ERROR recovery, SLEEP behavior, etc.)
- Consider breaking into sub-stories (US-0031a, US-0031b, etc.) if scope becomes too large
