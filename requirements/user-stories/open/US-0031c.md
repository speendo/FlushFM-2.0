# US-0031c: Remove RECOVER Handler (Placeholder for Error Recovery)

> **Status:** To Do | **Priority:** High | **Created:** 2026-05-09

## User Story
As a **maintainer**, I want **the untriggered RECOVER handler removed and replaced with a no-op placeholder**, so that **error recovery is explicitly marked as not-yet-designed rather than silently non-functional**.

## Acceptance Criteria
- [ ] `SystemEvent::RECOVER` enum value remains (needed for `toString()` and test compilation)
- [ ] ERROR handler at `supervisor.cpp:516`: `transitionTo(READY)` removed, RECOVER is a no-op (state unchanged)
- [ ] `test_state_transition_flow` RECOVER assertion updated: post RECOVER, expect state stays ERROR
- [ ] All other 63 tests pass unchanged; 4 pre-existing errors unchanged
- [ ] Debug and production builds succeed

## Notes

### Current handler

```cpp
// supervisor.cpp:515-519
case SystemState::ERROR:
    if (event == SystemEvent::RECOVER) {
        transitionTo(SystemState::READY, event, reason);
    }
    break;
```

### New handler

```cpp
// supervisor.cpp:515-519
case SystemState::ERROR:
    if (event == SystemEvent::RECOVER) {
        // Placeholder: error recovery is not yet designed.
        // When designed, RECOVER will trigger BOOTING → CONNECTING → READY → LIVE
        // sequential traversal with retry policy and FATAL exhaustion.
    }
    break;
```

### Test update

In `test/test_state_transition_flow/test_main.cpp`, function `test_error_state_transitions`:

**Before (line ~336):**
```cpp
TEST_ASSERT_TRUE(fixture.controller.postEvent(SystemEvent::RECOVER, SystemReason::RECOVERY));
TEST_ASSERT_EQUAL(static_cast<int>(SystemState::READY), static_cast<int>(fixture.controller.state()));
```

**After:**
```cpp
TEST_ASSERT_TRUE(fixture.controller.postEvent(SystemEvent::RECOVER, SystemReason::RECOVERY));
TEST_ASSERT_EQUAL(static_cast<int>(SystemState::ERROR), static_cast<int>(fixture.controller.state()));
```

This verifies RECOVER is a valid-to-post event (no crash) but doesn't change state. When recovery is designed, flip the assertion to the new target state.

### Why keep RECOVER in the enum?

Removing it from the X-macro would require updating `toString()` tests in `test_component_types`. Keeping it avoids cascading test changes for zero behavioral gain. RECOVER is harmless as a valid enum value with no active handler.

### Production impact

None. RECOVER was never posted in production code. Removing the handler changes nothing — the system was already stuck in ERROR with or without this handler.
