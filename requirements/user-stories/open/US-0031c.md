## Notes

### Current behavior (to change)

```cpp
case SystemState::ERROR:
    if (event == SystemEvent::RECOVER) {
        transitionTo(SystemState::READY, event, reason);  // SKIPS BOOTING+CONNECTING
    }
    break;
```

### New behavior

```cpp
case SystemState::ERROR:
    if (event == SystemEvent::RECOVER) {
        targetMode_ = SystemState::READY;  // ultimate goal
        transitionTo(SystemState::BOOTING, event, reason);  // restart at BOOTING
        // orchestration then drives BOOTING â†’ CONNECTING â†’ READY â†’ LIVE
    }
    break;
```

The `transitionTo(BOOTING)` call:
1. Sets `observedState_ = BOOTING`
2. Clears `transientError_`
3. Notifies observers
4. Then the caller (handleEvent or orchestration completion) checks: is `observedState_` at `targetMode_`? If not, start orchestration to the next state in sequence.

### Sequential traversal mechanism

After transitioning to BOOTING, the system must automatically advance through CONNECTINGâ†’READYâ†’LIVE. This requires a mechanism to detect "we reached a non-target state, keep going."

Simplest approach: after `transitionTo(BOOTING)` in the RECOVER handler, immediately call `requestStateTransition(nextInSequence(BOOTING))` which is CONNECTING. Then in reportCompletion, after reaching CONNECTING, if `observedState_ != targetMode_`, automatically request next state.

Helper function:
```cpp
SystemState nextLinearState(SystemState current) {
    switch (current) {
        case SystemState::BOOTING:   return SystemState::CONNECTING;
        case SystemState::CONNECTING: return SystemState::READY;
        case SystemState::READY:     return SystemState::LIVE;
        default: return current;  // LIVE, SLEEP, ERROR, FATAL â€” no next state
    }
}
```

In reportCompletion, after successful transition commit (target reached without failure):
```cpp
if (!orchestration_.requiredFailure && observedState_ == targetMode_) {
    // Check if we need to continue sequential traversal
    // Only for non-target completion â€” US-0031b handles the general case
}
```

Actually, the sequential traversal should be driven by the same mechanism as US-0031b's `targetMode_` system. When RECOVER fires:
1. `targetMode_ = READY`
2. `transitionTo(BOOTING)` â€” observedState_ = BOOTING
3. Since `observedState_ != targetMode_`, start orchestration to CONNECTING
4. When CONNECTING is reached, `observedState_ != targetMode_` â†’ start orchestration to READY
5. When READY is reached, `observedState_ == targetMode_` â†’ done (or if target was LIVE, continue)

This can be implemented in a post-transition hook in `transitionTo()` or in the RECOVER handler itself.
