# US-0014: Component Architecture – Extend ISystemComponent with State Methods and Compile-Time Safe Timeouts

> **Status:** To Do  
> **Priority:** High  
> **Created:** 2026-03-29

---

## User Story

As a **component developer**,  
I want **explicit setOFF, setIDLE, setSTREAMING, setERROR methods in ISystemComponent**,  
so that **the compiler forces me to implement all required state transitions**.

---

## Acceptance Criteria

- [ ] Update `src/system_components.h`: ISystemComponent interface extended
- [ ] Add virtual methods returning timeout in milliseconds: `uint32_t setOFF(uint32_t transitionId)`, `setIDLE(uint32_t transitionId)`, `setSTREAMING(uint32_t transitionId)`, `setERROR(uint32_t transitionId)`
- [ ] Add timeout hook method: `void onTransitionTimeout(uint32_t transitionId)`
- [ ] `setXXX()` timeout return is static component-owned configuration for that target state
- [ ] All existing concrete components (BoardInfo, WiFi, AudioRuntime, CLI) updated with implementations
- [ ] No compilation errors on all components
- [ ] At least one component (e.g., BoardInfoComponent) has real, complete implementations
- [ ] Timeouts are state-specific and compile-time enforced through required `setXXX()` methods
- [ ] Build configuration treats missing virtual overrides as compile errors for concrete components

---

## Test Strategy

**Write tests first (TDD per `testing.md`):**
- [ ] Unit test: Compile-check that concrete implementations must override all `setXXX()` methods → missing override = compiler error
- [ ] Unit test: `setOFF()`, `setIDLE()`, `setSTREAMING()`, `setERROR()` each return `uint32_t` timeout value (test stub implementations)
- [ ] Unit test: `onTransitionTimeout(uint32_t transitionId)` can be called and component safely moves to idle state
- [ ] Unit test: All four existing components (BoardInfo, WiFi, AudioRuntime, CLI) have all required implementations (no pure virtual left)
- [ ] Integration check: Build all components with strict warning-as-error enabled; no missing virtual method warnings

**Corner Cases & Runtime Safety:**
- [ ] Unit test: `setXXX(0)` (zero transitionId) is handled correctly (valid but edge case)
- [ ] Unit test: `setXXX(UINT32_MAX)` (max transitionId) does not cause overflow
- [ ] Unit test: Timeout return value of 0 is accepted (immediate completion expected)
- [ ] Unit test: Timeout return value of UINT32_MAX is accepted (very long timeout)
- [ ] Unit test: `onTransitionTimeout()` called twice with same ID doesn't crash component
- [ ] Unit test: Component in safe state after `onTransitionTimeout()` is verifiable (no dangling pointers, no resource leaks)
- [ ] Unit test: Multiple components' `setXXX()` calls do not interfere (state isolation)
- [ ] No memory leaks when calling `setXXX()` repeatedly
- [ ] No deadlock or infinite loop if component synchronously calls back during `setXXX()`

Tests run on host and validate interface contract before component development begins.

---

## Notes

Each component knows its own timing best. Timeout values are provided by each `setXXX()` method.
No controller fallback and no separate timeout lookup API.
After every `setXXX()` call, the component must emit exactly one completion callback (`Completed` or `Failed`).
On timeout, the SystemController calls `onTransitionTimeout(transitionId)`; the component must move to a safe internal state and then report `Failed` with an optional reason.
