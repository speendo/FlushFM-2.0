# US-0026a: Static Component Identifier Migration (Functional)

> **Status:** To Do | **Priority:** Medium | **Created:** 2026-04-19

## User Story
As a **maintainer**, I want **controller/component identity APIs to use static component identifiers**, so that **program logic no longer depends on runtime string names**.

## Acceptance Criteria
- [ ] Component names remain static debug/log labels and are not used as runtime program-logic keys
- [ ] Name-based controller APIs are migrated to static component identifiers (for example `ComponentId`) for registration, hook binding, status lookup, and completion reporting; in-flight transition tracking remains orchestration-managed
- [ ] `markComponentFailed(...)` is removed from the public controller API (or reduced to an internal-only helper) and replaced by normal failure signaling through transition completion/events
- [ ] `registerComponent(...)` is simplified to static identifier-based direct access (no `normalizeComponentName(...)`, no empty-name guard path, no string-map lookup path)
- [ ] `beginComponentTransition(...)` is removed from the public controller API; orchestration-managed in-flight tracking remains the only production path
- [ ] `pendingTransitions_` is orchestration-owned state only: no pending component-transition entries exist when `orchestration_.active == false`
- [ ] `reportCompletion(...)` is simplified to static identifier-based direct access (no `normalizeComponentName(...)`, no empty-name guard path, no string-map lookup path)
- [ ] Public controller methods that only return an internal boolean member are kept only if that boolean remains part of the production contract; obsolete passthrough getters are removed from the API
- [ ] `getComponentStatus(...)` is simplified to static identifier-based direct access (no `normalizeComponentName(...)`, no empty-name guard path, no string-map lookup path)
- [ ] `isComponentRequired(...)` is simplified to static identifier-based direct access (no `normalizeComponentName(...)`, no empty-name guard path, no string-map lookup path)
- [ ] Component identity lookup remains deterministic and continues to support registration, hook binding, in-flight transition tracking, and completion correlation
- [ ] In-flight transition entries are guaranteed to belong to the active orchestration transition id; redundant mismatch-guard branches in timeout scanning are removed once this invariant is enforced
- [ ] Hook binding is static and deterministic; runtime null-callback and unknown-component guard branches in `setComponentTransitionHooks(...)` are removed
- [ ] No separate runtime behavior is introduced for test vs production; logic remains single-path and deterministic
- [ ] Graceful degradation behavior is preserved exactly: optional or missing dependencies must not crash the system or block boot
- [ ] Unit tests are updated to cover identifier-based registration/lookup/completion behavior
- [ ] Unit tests are migrated away from direct `beginComponentTransition(...)` usage to production-equivalent orchestration/completion flows
- [ ] Unit tests that currently depend on `markComponentFailed(...)` are migrated to the new failure-signaling path
- [ ] Debug and production builds succeed

## Notes
- This story is functional migration only.
- Static component names remain available for logs, diagnostics, and human-readable output.
