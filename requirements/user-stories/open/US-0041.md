# US-0041: Lazy / Busy Transition Mode

> **Status:** To Do | **Priority:** Medium | **Created:** 2026-05-17

## User Story
As a **maintainer**, I want **the supervisor to tell components whether to use lazy or busy transition mode on each state request**, so that **components skip unnecessary work when already in a valid state range (e.g., display stays on during ERROR recovery)**.

## Acceptance Criteria
- [ ] Supervisor sends a lazy/busy indicator alongside each target state written to component mailboxes
- [ ] Lazy: minimal steps — if component is already within its valid range for the target system state, stay put; if out of range, reach the nearest boundary
- [ ] Busy: best effort — go as far as possible in the direction of movement (upward: reach maxState; downward: reach minState)
- [ ] Display component, LIVE→ERROR, range [ERROR, LIVE], lazy: stays at LIVE → display stays on (range is only an example)
- [ ] Display component, LIVE→ERROR, range [ERROR, LIVE], busy: strives for ERROR → display tries to turn off (range is only an example)
possible)
- [ ] Debug and production builds succeed
- [ ] Native unit tests pass
