# US-0042: Flexible Transition Timeouts (Component-Driven)

> **Status:** Idea | **Priority:** Low | **Created:** 2026-05-17

## User Story
As a **maintainer**, I want **transition timeouts to be driven by component estimates rather than a static supervisor-level matrix**, so that **per-transition timing matches actual component needs and avoids false timeouts**.

## Acceptance Criteria
- [ ] Components return timeout estimates from their `setXXX()` methods (currently returned but ignored)
- [ ] Supervisor uses `max(all component timeouts)` as the orchestration deadline
- [ ] `TransitionTimeoutConfig` matrix is removed — timings come from components, not supervisor configuration
- [ ] Debug and production builds succeed
- [ ] Native unit tests pass

## Notes
- Full design in `docs/superpowers/specs/2026-05-17-supervisor-architecture-freeze.md` §3.2
- **Unresolved doubt:** For truly open-ended operations (e.g., WiFi connection), the component cannot predict how long it will take. The cap-based model (US-0040) may cover this case by letting components signal "I need more time" via a lowered cap rather than an extended timeout.
- Low priority — US-0040 (caps) likely covers the motivating use case. This story is kept as a design alternative.
- If implemented: replaces `TransitionTimeoutConfig`, does not augment it.
