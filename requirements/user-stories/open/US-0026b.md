# US-0026b: Runtime Ballast Removal After Identifier Migration

> **Status:** To Do | **Priority:** Medium | **Created:** 2026-04-19

## User Story
As a **maintainer**, I want **runtime string-processing and defensive ballast removed after static identifier migration**, so that **the controller path is lightweight for ESP32 without changing behavior**.

## Acceptance Criteria
- [ ] `normalizeComponentName(...)` is removed from `SystemController`
- [ ] Name-length truncation and related logging are removed from runtime controller paths
- [ ] Name-validation branches for null/empty/too-long component names are removed from migrated identifier-based controller methods
- [ ] Failure handling remains independent from free-form reason text for control-path decisions
- [ ] `copyFailureReason(...)` and mutable runtime reason-buffer copying are removed from controller hot paths
- [ ] Failure diagnostics are represented via stable static semantics (for example status + error code or static literals), not dynamic text processing
- [ ] No separate runtime behavior is introduced for test vs production; logic remains single-path and deterministic
- [ ] Graceful degradation behavior is preserved exactly: optional or missing dependencies must not crash the system or block boot
- [ ] Unit tests are updated for the cleaned-up controller behavior and diagnostics semantics
- [ ] Debug and production builds succeed

## Notes
- This story depends on the functional identifier migration from `US-0026a`.
- Scope is cleanup/performance only; no functional broadening.
