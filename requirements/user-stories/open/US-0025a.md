# US-0025a: Event Model Contract and Naming Decisions

> **Status:** To Do | **Priority:** High | **Created:** 2026-04-19

## User Story
As a **maintainer**, I want **the event model and naming contract to be defined before code migration starts**, so that **the refactor can be implemented consistently and without ambiguity**.

## Acceptance Criteria
- [ ] SystemController remains the single source of truth for global system state transitions (`BOOTING`, `SLEEP`, `CONNECTING`, `READY`, `LIVE`, `ERROR`)
- [ ] Components may keep additional internal states beyond system states without exposing low-level details as system events
- [ ] Each `SystemState` defines explicit guarantees for required component states; components mirror those guarantees in their own lifecycle state machines instead of pushing component-specific state logic into `SystemController`
- [ ] Components never set global system state directly; they only emit normalized domain events to SystemController
- [ ] `SystemEvent` uses a minimal generic contract: external state requests use `STATE_REQUESTED` (with target-state payload), while `STATE_ENTERED` and `STATE_FAILED` remain internal orchestration outcomes
- [ ] All events are treated uniformly: no special deferred intent buffering; FIFO queue processing with transition context guards is the sole ordering mechanism
- [ ] Readiness source is unified: `READY` eligibility is derived from component lifecycle status (for required components) and no separate startup read flags are used
- [ ] Queue send failure contract is explicit: at minimum, every failed event enqueue is logged with `ERROR_LOG` including event context
- [ ] `EventPolicy` is renamed to clearer semantics (for example `BestEffort` and `Critical`) and the intended post-event handling semantics are documented
- [ ] `SystemReason` is migrated to static text metadata (`const char*` with static lifetime) and is limited to logging/debug context; it is not used as a causal input for state-transition decisions
- [ ] `postEvent(...)` reason contract is explicit and documented: reason metadata uses static-lifetime text only, is optional for logs/debugging, and must not alter transition arbitration logic
- [ ] Enum names and `toString(...)` labels are derived from a single source of truth (for example an X-macro list or equivalent central mapping) so that string labels are not duplicated across separate switch statements
- [ ] A normalized event contract is defined and documented for each component: `local signal -> domain event -> expected normalized system effect`
- [ ] Existing event ordering and arbitration are described deterministically without a static priority table: queue order plus transition-context guards are the intended model
- [ ] Queue-processing naming is explicit: the queue-drain entrypoint is named `processEventQueue()` (replacing ambiguous `dispatchPending()` wording)
- [ ] Naming is consistent across declaration, implementation, and CLI/debug pathways (no mixed aliases for equivalent events)
- [ ] Open design decisions are recorded in the story notes before implementation begins

## Notes
This story is contract-first. It is complete when the event taxonomy, naming changes, and the reason-text contract are agreed and documented.

### [STATE GUARANTEE MODEL]
The global system state defines the guarantees that each component must satisfy. Component lifecycle states mirror those guarantees locally. `READY` is the state name used in this story and in the current codebase for the user-facing idle/ready baseline.

| SystemState | Component guarantees |
| --- | --- |
| `BOOTING` | Start state for all components; WiFi is not connected; Audio may be uninitialized; components are still allowed to initialize themselves |
| `SLEEP` | All components are initialized; WiFi is not connected; Audio is muted; CLI is ready; all components are in their lowest-power mode |
| `CONNECTING` | WiFi is attempting to connect; Audio is muted; all other components are at least in `SLEEP` or above |
| `READY` | WiFi is connected; Audio is muted; all other components satisfy the same baseline guarantees as `SLEEP` |
| `LIVE` | All components are active |

### [ARCHITECTURE DECISIONS]
- [DECIDED] **No Intent/Domain Event Distinction**: With light-switch-based control, all events are equivalent in priority. There is no need to buffer certain "intent" events while dropping others. All events are processed FIFO from a single queue with transition-context guards. This simplifies `postEvent()` and removes `isIntentEvent()` and `deferredIntentEvents_` entirely.
- [DECIDED] **Single Readiness Source**: `READY` is derived from required component lifecycle status. Separate startup read flags (`startupWiFiReady_`, `startupAudioReady_`) are removed to avoid duplicate readiness truth.
- [DECIDED] **Mirrored Component Guarantees**: Global system states describe the guarantees each component must satisfy, and component lifecycle states mirror those guarantees locally. `handleEvent()` should remain a state-orchestration dispatcher, not the place where component-specific state machines are encoded.
- [DECIDED] **Single Source for Enum Names**: Enum names and `toString(...)` outputs should be derived from one shared definition (for example an X-macro list or equivalent central mapping) to avoid duplicate string maintenance across multiple switches.
- [DECIDED] **Function-Level Scope Is Explicit**: Contract decisions in this story directly govern `postEvent()`, `processEventQueue()` (renamed from `dispatchPending()`), `handleEvent()`, and the removal of `isIntentEvent()`.
- [DECIDED] **Transition Outcomes Stay Internal**: Transition outcomes remain internal orchestration outcomes and are not introduced as additional `SystemEvent` values.
- [DECIDED] **Diagnostics Stay Out of State Events**: Component diagnostics remain in logging/telemetry channels and are not posted as state-driving `SystemEvent` values.
- [DECIDED] **One-Pass Naming Migration**: Naming/event migration is executed in one clean pass without long-lived compatibility aliases.
- [DECIDED] **Queue Failure Escalation Deferred**: `ERROR_LOG` is the mandatory minimum reaction for enqueue failure in this refactor; hard escalation policy is handled in a dedicated follow-up decision.
- [DECIDED] **Minimal Generic Event Contract**: Controller-facing events are reduced to a minimal generic model: `STATE_REQUESTED` for external state intents, while `STATE_ENTERED` and `STATE_FAILED` are internal orchestration outcomes. Legacy specific event names are mapped through component/local adapters and are not the long-term controller contract.
- [DECIDED] **SystemReason Uses Static Diagnostic Text**: `SystemReason` is represented as static text metadata (`const char*`, static lifetime) for logging/debugging only. It does not drive state logic, transition guards, or arbitration behavior.
