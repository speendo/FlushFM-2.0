# US-0021: CLI Stream Command Simplification

> **Status:** To Do  
> **Priority:** Medium  
> **Created:** 2026-04-02

---

## User Story

As a **developer**,  
I want **the stream-related CLI commands to use a single canonical flow**,  
so that **the command set is easier to understand, easier to maintain, and less likely to diverge over time**.

---

## Acceptance Criteria

- [ ] The current behavioral difference between `play` and `switch` is documented in the story and reflected in the CLI help text
- [ ] `play [url]` becomes the only user-facing stream command and continues to support the optional last-saved station fallback
- [ ] `switch <url>` is removed from the supported CLI surface and no longer appears in help, usage, or tests
- [ ] The remaining stream command uses one shared underlying path so parsing, persistence, and error handling cannot drift apart
- [ ] Command output clearly describes `play` as the single start/resume command
- [ ] Existing persistence rules remain unchanged: station URLs are still saved when a stream is started via `play`
- [ ] Native tests cover the `play` command path and verify the stream setup side effects
- [ ] Debug and production builds succeed after the command alignment change

---

## Notes

### Current Difference Between the Methods

- `play [url]` can be called with or without a URL.
- If no URL is provided to `play`, the CLI uses the last saved station from NVS.
- `switch <url>` currently requires an explicit URL and behaves as a separate command for replacing the active stream with another station.
- Today both commands end up starting stream playback, but they are presented as separate user-facing actions in help text and output messages.
- The simplification target is to keep `play` and remove `switch` from the supported command set.

### Adjustment Plan

1. Keep `play [url]` as the canonical command for starting playback and for reusing the last persisted station.
2. Remove `switch <url>` from the CLI parser, help text, usage messages, and tests.
3. Route all stream-starting behavior through the `play` command path to avoid duplicated parsing and behavior drift.
4. Update error messages and documentation so users understand that `play` is now the single entry point for starting or changing streams.
5. Add or update tests for the `play` path to verify that stream selection, persistence, and logging remain consistent after `switch` removal.
