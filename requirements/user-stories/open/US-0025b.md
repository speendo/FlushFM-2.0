# US-0025b: Event Model Migration, Cleanup, and Tests

> **Status:** To Do | **Priority:** High | **Created:** 2026-04-19

## User Story
As a **maintainer**, I want **the event-model changes to be migrated through the codebase and verified by tests**, so that **the refactor is safe, complete, and behaviorally stable**.

## Acceptance Criteria
- [ ] `SystemEvent` call sites are migrated to the agreed minimal generic contract from `US-0025a` (`STATE_REQUESTED` externally; `STATE_ENTERED`/`STATE_FAILED` internal orchestration outcomes)
- [ ] `isIntentEvent()` helper and all deferred-intent logic are completely removed; all events are processed uniformly from the single FIFO queue
- [ ] The following legacy members are removed with their associated code paths: `deferredIntentEvents_`, `pendingCriticalEvent_`, `pendingEvent_`, `pendingReason_`, `pendingReplayRequested_`, `deferredReplayEvent_`, `pendingPlayAfterReady_`, `deferredPlayAfterReadyEvent_`, `startupWiFiReady_`, `startupAudioReady_`
- [ ] Any public/private methods that only expose removed boolean members are removed as dead API surface; boolean passthrough getters remain only for booleans that are still required after migration
- [ ] `postEvent()` is simplified: no special intent handling logic; all events follow the same queue path with uniform EventPolicy backpressure handling
- [ ] On enqueue failure in `postEvent()`, `ERROR_LOG` is emitted unconditionally with event context as the minimum failure reaction
- [ ] Harder enqueue-failure handling (automatic restart, explicit ERROR-state transition, or equivalent) is intentionally out of scope for this story and documented as a follow-up decision
- [ ] `processEventQueue()` (renamed from `dispatchPending()`) is simplified: no intent-specific deferral logic; queue drain + transition-context guards + timeout checks remain
- [ ] `handleEvent()` is simplified to remove intent-specific branching paths that depend on deferred intent buffering
- [ ] `handleEvent()` is reduced to a deterministic state/event transition mapping with explicit guard clauses; transition side effects are moved into small helper paths to avoid monolithic nested branching
- [ ] `handleEvent()` no longer depends on removed replay/readiness/deferred boolean members (`pendingReplayRequested_`, `deferredReplayEvent_`, `pendingPlayAfterReady_`, `deferredPlayAfterReadyEvent_`, `startupWiFiReady_`, `startupAudioReady_`)
- [ ] For each accepted `SystemEvent` in each `SystemState`, exactly one normalized outcome is documented and test-covered (`state unchanged`, `request transition`, or `direct transition`)
- [ ] Legacy specific event names are mapped through adapter paths to the generic controller contract and are removed from the long-term controller event surface
- [ ] `transitionTo()` is reduced to a small internal commit helper that applies the new global state, logs the transition, and notifies observers; replay/deferred cleanup and component-specific state logic move out of the function
- [ ] `isIntentEvent()` is removed entirely and no remaining call site depends on intent classification
- [ ] `READY` eligibility uses required component lifecycle status as the single readiness source; no duplicated readiness tracking flags remain
- [ ] `EventPolicy` call sites are migrated to the new names and queue/backpressure behavior remains functionally equivalent
- [ ] `SystemReason` is migrated to static diagnostic text metadata (`const char*`, static lifetime) and is used for logging/debug output only; no transition decision, guard, or arbitration logic depends on reason text
- [ ] `toString(...)` helpers for enums are refactored to use the shared single-source enum/string definition from `US-0025a` so that string labels are not duplicated across separate switch bodies
- [ ] Long-lived controller state is reorganized into small cohesive groups where this improves readability (for example queue/recovery state and orchestration state), without changing behavior
- [ ] Short-lived helper variables remain local to methods; only persistent cross-call state stays as controller members
- [ ] Existing callback type aliases (`StateObserver`, `TransitionInvoker`, `TransitionTimeoutHook`) remain as the stable component/controller contract unless a direct simplification is proven beneficial
- [ ] A per-component mapping table is implemented or updated in code/comments where needed: `local signal -> system event -> expected normalized system effect`
- [ ] Existing event ordering and arbitration remain deterministic: events are processed in queue order, while stale or invalid transition outcomes are ignored via transition context checks (`transitionId`, active transition state, single-completion guard)
- [ ] Native unit tests cover component signal mapping and controller arbitration with stale/duplicate event safety
- [ ] Debug and production builds succeed

## Notes
This story is implementation-first. It should not broaden the contract; it should apply the decisions from `US-0025a` and validate them in code.

Function-level simplification focus in this story:
- `postEvent()`: remove sticky deferred-intent fallback paths and keep uniform enqueue behavior with mandatory `ERROR_LOG` on failure.
- `processEventQueue()`: keep queue draining responsibilities explicit and remove intent-classification branches.
- `handleEvent()`: remove event-classification-dependent branches tied to deferred intent buffering.
- `transitionTo()`: keep only the minimal commit steps for a global state change; do not let it own replay/deferred recovery behavior or component-local state mirroring.
- `toString(...)`: derive enum names from the same shared definition used by the enums themselves, rather than maintaining separate per-enum switch tables.
