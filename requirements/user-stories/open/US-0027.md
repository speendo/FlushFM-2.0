# US-0027: Supervisor as a Module, Not a Component

> **Status:** To Do | **Priority:** Medium | **Created:** 2026-04-19

## User Story
As a **maintainer**, I want **the Supervisor to be treated as an infrastructure module instead of a component**, so that **its responsibilities, API, and logging remain clearly separated from component lifecycle management**.

## Acceptance Criteria
- [ ] The Supervisor is modeled as a module/orchestrator, not as a component participating in the component registry
- [ ] The Supervisor does not register itself through the component lifecycle API and does not require a component identity for program logic
- [ ] The public Supervisor API is explicit and module-oriented, covering Mailbox processing, event posting, orchestration, and transition inspection without exposing unnecessary internal state
- [ ] Component lifecycle management remains limited to actual runtime components such as WiFi, AudioRuntime, CLI, and BoardInfo
- [ ] Supervisor dispatches global state guarantees to components; components own their mirrored lifecycle states and readiness semantics instead of encoding those state machines inside the controller
- [ ] `transitionTo()` remains an internal commit helper for module-owned global state changes; it is not exposed as part of the public module boundary
- [ ] The Supervisor keeps its own static debug/log source label only for diagnostics and logging
- [ ] Component transition hooks are treated as statically wired module contracts, not dynamic runtime configuration points
- [ ] Runtime guard branches in hook setup that contradict static invariants (for example unknown component and missing callback checks) are removed from production paths and enforced through deterministic initialization plus tests/debug assertions
- [ ] Module boundaries are documented so that future refactors do not reintroduce controller-as-component coupling
- [ ] Unit tests are updated or added where needed to ensure controller/module behavior remains unchanged after the boundary clarification
- [ ] Debug and production builds succeed

## Notes
- This story is about architectural boundary clarity, not behavior changes.
- It should align with the decisions from US-0025h and the static-identifier cleanup in US-0026f.
- Graceful degradation requirements still apply; the controller must remain robust if optional subsystems are unavailable.
- Implementation scope is split into fine-grained stories: US-0027a, US-0027b, US-0027c.
