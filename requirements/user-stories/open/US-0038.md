# US-0038: Persist Supervisor Configuration to NVS

> **Status:** To Do | **Priority:** Medium | **Created:** 2026-05-14

## User Story
As a **developer**, I want `maxRecoveries` and the transition timeout matrices to be persisted to NVS and restored on boot, so that recovery policy and transition timing configuration survive device reboots.

## Acceptance Criteria

### Load from NVS
- [ ] `SupervisorV2::setup()` loads `maxRecoveries` from NVS after system initialization
  - [ ] If NVS key is missing or invalid, default to `3`
  - [ ] Loaded value is applied via `setMaxRecoveries()`

### Save to NVS
- [ ] `SupervisorV2::setMaxRecoveries()` validates input (≥ 1) and updates the member variable
  - [ ] After validation, writes the new value to NVS
  - [ ] Invalid values (< 1) are rejected and logged; NVS is not updated

### Transition Timeout Matrices
- [ ] `SupervisorV2::setup()` loads the complete transition timeout matrix from NVS in one operation
  - [ ] If the NVS blob is missing or invalid, initialize with sensible defaults and log a warning
  - [ ] The loaded matrix is applied to the Supervisor timeout config as a whole
- [ ] The transition timeout matrix is saved to NVS as a single logical block
- [ ] The story does not require implementation of the timeout matrix persistence yet; it only reserves the requirement for the follow-up implementation

### Validation
- [ ] Native unit tests verify load/save round-trip for maxRecoveries
- [ ] Native unit tests verify load/save round-trip for the transition timeout matrix
- [ ] Native tests verify default behavior when NVS is empty
- [ ] No NVS corruption on invalid input attempts
- [ ] `pio test -e native` passes

## Notes
- Prepare infrastructure to call `settings::load()` and `settings::save()` from Supervisor; see `lib/settings/` for existing NVS wrapper
- Error handling: log failures but don't halt; proceed with defaults if load fails
- Consider batching NVS writes if `setMaxRecoveries()` is called frequently during recovery attempts
- Timeout matrices are part of the persistence scope, but their implementation is intentionally deferred for now.
