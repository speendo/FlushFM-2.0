# US-0040: Component-Driven Maximum Achievable State (Cap)

> **Status:** To Do | **Priority:** Medium | **Created:** 2026-05-17

## User Story
As a **maintainer**, I want **the supervisor to respect a per-component maximum achievable state cap**, so that **the system can gracefully wait for resolvable conditions (e.g., missing WiFi credentials) without escalating to ERROR→FATAL recovery**.

## Acceptance Criteria
- [ ] Components can signal their current state ceiling via `setCap(id, SystemState)` (push-based, no polling)
- [ ] Supervisor stores per-component caps and computes `systemCap = min(all component caps)`
- [ ] When `targetState <= systemCap`: normal orchestration toward target
- [ ] When `targetState > systemCap`: supervisor steps toward `systemCap` as effective target, preserving original intent in `targetState_` (last-write-wins) for when cap rises
- [ ] When `systemCap` drops below `observedState`: supervisor triggers a downward step toward the new cap
- [ ] Downward state requests are always allowed, regardless of cap
- [ ] Components signal cap cleared/unblocked via `setCap(id, HIGHER_STATE)` — no polling from either side
- [ ] WiFi component without credentials sets cap to `CONNECTING` → system does not ERROR out
- [ ] Debug and production builds succeed
- [ ] Native unit tests pass
