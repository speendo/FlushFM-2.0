# US-0003: Core Pinning Verification

> **Status:** Done  
> **Priority:** High  
> **Created:** 2026-03-01

---

## User Story

As a **developer**,  
I want **to verify that ESP32-audioI2S can be effectively pinned to Core 1 via custom task wrapper**,  
so that **I can implement the dual-core architecture with confidence that audio tasks won't interfere with Core 0 system tasks**.

---

## Acceptance Criteria

*Each criterion should be testable and unambiguous. A story is "done" when all criteria are met.*

- [x] `audioTask` is pinned to Core 1 via `xTaskCreatePinnedToCore` (`AUDIO_TASK_CORE = 1`)
- [x] Audio streaming works correctly when `audio.loop()` is called from the Core 1 task
- [x] Task monitoring (`tasks` Serial command) confirms `audioTask` runs exclusively on Core 1
- [x] Task monitoring documents all tasks visible on Core 0; framework-internal tasks (WiFi/TCP, I2S DMA, IDLE) are expected and accepted per `concurrency.md` exception – no application-level audio logic runs on Core 0
- [x] Audio quality on Core 1 is at least as good as previously observed on Core 0
- [x] Audio playback remains stable under simulated Core 0 load (busy-loop task via `loadtest` Serial command)
- [x] After suspend/resume of `audioTask`, audio resumes cleanly (no reboot, no crash, no permanent silence)
- [x] Stack high-water mark of `audioTask` is acceptable (≥ 512 free DWORDs reported by monitoring)

---

## Notes

Requires US-0002 completed. Task wrapper pattern → `concurrency.md` Reference Pattern.

**Verify with `uxTaskGetSystemState()` via `tasks` Serial command:**
- `audioTask` → Core 1
- Framework tasks on Core 0 (`wifi`, `timerTask`, `IDLE0`, etc.) are not created by our code and cannot be pinned – accepted per `concurrency.md`: *"Internal library-managed tasks are exempt from manual pinning rules"*
- Check `audioTask` stack high-water mark for overflow risk

**Suspend/Resume scope:** Dropouts during suspend are expected (I2S buffer drains). The criterion is that the system recovers cleanly after `vTaskResume()` – no watchdog, no reboot, no locked I2S state.

**Observed DMA drain behavior:** After `vTaskSuspend()`, audio continues playing for several seconds because the I2S DMA hardware drains its already-filled buffers independently of the CPU task. This is expected and by design. In production, this means audio will not stop immediately when the system triggers a sleep/relay-off – a clean stop sequence (`audio.stopSong()` before cutting power) is required. This is tracked in **US-0007**.
