# US-0003: Core Pinning Verification

> **Status:** To Do  
> **Priority:** High  
> **Created:** 2026-03-01

---

## User Story

As a **developer**,  
I want **to verify that ESP32-audioI2S can be effectively pinned to Core 1 via custom task wrapper**,  
so that **I can implement the dual-core architecture with confidence that audio tasks won't interfere with Core 0 system tasks**.

---

## Acceptance Criteria

*Each criterion should be testable and unambiguous. A story is "done" when all criteria are met.*

- [ ] Custom `audioEngineTask` successfully created and pinned to Core 1 using `xTaskCreatePinnedToCore`
- [ ] Audio streaming works correctly when `audio.loop()` is called from the Core 1 task
- [ ] Task monitoring confirms the main audio task runs exclusively on Core 1
- [ ] Task monitoring shows no unexpected audio-related tasks created on Core 0
- [ ] Performance comparison: custom wrapper vs library default (CPU usage, audio quality)
- [ ] Audio playback remains stable under simulated Core 0 load (busy loop test)
- [ ] Core 1 task can be suspended/resumed without audio corruption
- [ ] Memory usage patterns are acceptable (no excessive stack/heap consumption)

---

## Notes

*This test validates our critical architectural assumption. If ESP32-audioI2S creates internal tasks that ignore our Core 1 wrapper, we need to know before building the full system.*

**Test approach:**
1. Start with basic audio working (US-0001 completed)
2. Implement the custom task wrapper pattern:
   ```cpp
   void audioEngineTask(void *pvParameters) {
       while(1) {
           audio.loop(); 
           // NO delay() - let audio.loop() manage timing
       }
   }
   
   xTaskCreatePinnedToCore(audioEngineTask, "AudioEngine", 10000, NULL, 3, NULL, 1);
   ```
3. Use `uxTaskGetSystemState()` to monitor all tasks and their core assignments
4. Compare performance vs letting library manage its own tasks

**What to watch for:**
- Hidden tasks created by HTTP client, audio decoder, I2S driver
- Task priorities causing audio interruption 
- Stack overflow in the wrapper task
- PSRAM buffer management delays

**Success criteria:**
If all audio-related CPU activity stays on Core 1, we can proceed with the dual-core architecture. If not, we may need to reconsider the library choice or accept mixed-core operation.
